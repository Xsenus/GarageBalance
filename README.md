# GarageBalance

Веб-система финансового учета для гаражно-строительного кооператива.

Стартовый HTML приложения размечен как русскоязычный (`lang="ru"`), а вкладка браузера называется `GarageBalance - учет ГСК`, чтобы тестовый стенд и будущая production-сборка не выглядели как шаблонный Vite/frontend-проект.

## Стек

- Backend: C# / ASP.NET Core.
- Frontend: React + TypeScript + Vite.
- Database: PostgreSQL.
- Packaging: Docker Compose подготовлен заранее, финальная упаковка планируется ближе к сдаче.

## Структура

- `backend/GarageBalance.Api` - API, конфигурация, будущая бизнес-логика и `AppReleases`.
- `backend/GarageBalance.Api.Tests` - тесты backend.
- `frontend` - React-приложение.
- `docs` - roadmap, анализ исходных материалов, решения и история работ.
- `docker-compose.yml` - локальный запуск Postgres/API/frontend через Docker.

## Локальная разработка

Backend:

```powershell
dotnet tool restore
dotnet restore .\GarageBalance.slnx
dotnet tool run dotnet-ef database update --project .\backend\GarageBalance.Api\GarageBalance.Api.csproj --startup-project .\backend\GarageBalance.Api\GarageBalance.Api.csproj
dotnet run --project .\backend\GarageBalance.Api\GarageBalance.Api.csproj
```

Frontend:

```powershell
cd .\frontend
npm install
npm run dev
```

Для проверки production-сборки frontend:

```powershell
npm run build
npm run check:bundle
```

`npm run check:bundle` проверяет gzip-бюджет собранных JS/CSS assets: основной JS до 180 KiB, основной CSS до 40 KiB и общий JS/CSS budget до 260 KiB. Если добавляется тяжелая библиотека, превышение должно быть осознанным решением и фиксироваться в roadmap history.

- `GET /api/finance/summary` - итоги поступлений, выплат и баланса.
- `POST /api/finance/income` - внесение поступления по гаражу.
- `POST /api/finance/expense` - внесение выплаты поставщику.
- `POST /api/finance/operations/{id}/cancel` - отмена поступления или выплаты с обязательной причиной и audit-событием.

- `GET /api/import/access/runs/{id}/report` - скачивание JSON-отчета конкретного dry-run импорта.

- `PATCH /api/import/access/quarantine/{id}/resolve` - закрытие разобранной строки карантина с audit-событием.
- `POST /api/import/access/dry-run` - загрузка `.accdb`/`.mdb` и первичная проверка перед импортом.

Экран импорта Access показывает выбранный dry-run как наблюдаемую проверку: статус запуска, количество успешных проверок, предупреждений и ошибок, таблицу checks, пошаговый лог запуска и кнопку скачивания JSON-отчета. Пользовательские статусы выводятся по-русски: `Завершен`, `Заблокирован`, `Пройдено`, `Предупреждение`, `Ошибка`. История запусков показывает не только долю пройденных проверок, но и счетчики предупреждений/ошибок, например `2/3 · 1 предупреждение · 0 ошибок`. История, лог и карантин загружаются с явным серверным `limit`, чтобы панель импорта оставалась быстрой на длинной истории проверок и проблемных строк. Компактные таблицы лога, истории и карантина показывают счетчики `Показано X из Y`, если загружено больше строк, чем выводится на экран. Лог хранится в `access_import_run_log_entries`: UI получает безопасные шаги `file_received`, `hash_calculated`, `check_*`, `dry_run_finished`, уровень и сообщение, а служебный `DetailsJson` остается только в БД.

Для будущего фактического переноса Access добавлен реестр идемпотентности `access_import_row_fingerprints`: строка импорта регистрируется по источнику, типу сущности, внешнему id или SHA-256 row hash. Повторная регистрация возвращает существующий fingerprint и не должна создавать второй гараж, владельца, платеж или показание.

Проблемные строки будущего переноса Access складываются в `access_import_quarantine_items`: сервис хранит причину, уровень `error`/`warning`, SHA-256 row hash и JSON-снимок строки в БД, но публичные DTO показывают только безопасные поля без raw snapshot. В панели импорта администратор видит открытый карантин и может закрыть разобранную строку, действие фиксируется в audit.

- `GET /api/audit/events?limit=50` - audit-журнал действий пользователей и системных операций с серверным лимитом до 500 записей.
- `GET /api/audit/events/export` - CSV-экспорт текущей выборки audit-журнала с теми же фильтрами и маскированием чувствительных значений.
- `GET /api/reports/consolidated?limit=12` - консолидированный отчет за период с итогами по месяцам и гаражам; экран может передать `limit`, чтобы получить только видимые гаражные строки при сохранении полного `garageRowCount` и итогов.
- `GET /api/reports/consolidated/export/xlsx` - XLSX-выгрузка консолидированного отчета за период.
- `GET /api/reports/consolidated/export/pdf` - PDF-выгрузка консолидированного отчета за период.
- `GET /api/reports/income` - отчет по поступлениям: начисления, оплаты, фильтры по датам, поиску, гаражам, владельцам и видам поступлений; экран может передать `limit`, чтобы получить только видимые строки при сохранении полного `rowCount` и итогов.
- `GET /api/reports/income/export/xlsx` - XLSX-выгрузка отчета по поступлениям с теми же фильтрами, что экранный отчет.
- `GET /api/reports/income/export/pdf` - PDF-выгрузка отчета по поступлениям с теми же фильтрами, что экранный отчет.
- `GET /api/reports/expense` - отчет по выплатам: поставщики, виды выплат, документы, период и режим строк; экран может передать `limit`, чтобы получить только видимые строки при сохранении полного `rowCount` и итогов.
- `GET /api/reports/expense/export/xlsx` - XLSX-выгрузка отчета по выплатам с теми же фильтрами, что экранный отчет.
- `GET /api/reports/expense/export/pdf` - PDF-выгрузка отчета по выплатам с теми же фильтрами, что экранный отчет.
- `GET /api/app-releases` - история обновлений для раздела "Что нового" после входа в систему.

При первом применении миграций чистая PostgreSQL-БД получает системные виды поступлений и выплат из ТЗ. Виды поступлений: вода, мусор, электроэнергия, членский, целевой и вступительный взносы, подключения, штраф и предписание. Виды выплат: электроэнергия, вывоз мусора, водоснабжение, банковские, юридические, зарплата, прочие расходы и штрафы. Эти записи создаются с `IsSystem=true`, не удаляются как обычные пользовательские значения и не дублируются, если название уже было заведено вручную.

Длинные экранные отчеты показывают, что таблица обрезана до рабочего лимита: для гаражей выводится `Показано 12 из N строк`, для поступлений и выплат - `Показано 16 из N строк`. Экранные запросы сводного отчета, поступлений и выплат передают backend `limit`, но итоговые суммы и счетчики `garageRowCount`/`rowCount` считаются по полной выборке. Эти счетчики размечены `aria-live="polite"`, чтобы изменение количества видимых строк было доступно screen reader без ручного поиска по таблице.

На узкой ширине рабочая область перестраивается в одну колонку, строки таблиц становятся вертикальными, а основная навигация превращается в горизонтально прокручиваемую полосу. Для длинных значений в строках таблиц задано `min-width: 0`, чтобы текст переносился внутри ячейки и не растягивал экран телефона.

Экран отчетов сохраняет последние примененные фильтры сводного отчета, поступлений и выплат в `sessionStorage` текущей вкладки. При повторном открытии рабочей области первый запрос отчетов использует эти значения; испорченные или устаревшие данные sessionStorage игнорируются и заменяются дефолтами текущего периода.

В справочниках есть быстрый поиск гаражей по номеру или ФИО владельца и поиск поставщиков по названию, ИНН или контактному лицу. Оба поиска обращаются к backend-фильтрам, автоматически запускаются после короткой паузы при вводе и показывают счетчик найденных записей; кнопка рядом с полем остается для ручного запуска. Если очистить поле поиска, интерфейс автоматически возвращает полный список гаражей или поставщиков и показывает статус сброса фильтра.

Стартовый баланс гаража учитывается как начальная задолженность в отчетах по поступлениям и сводном отчете; стартовый баланс поставщика учитывается как начальное обязательство в отчете по выплатам. Для гаражей добавлена migration `GarageStartingBalance`.

Если расчет задолженности гаража уходит ниже нуля, frontend показывает это как "Переплата" и выводит сумму без минуса. Исходное поле API `debt` остается отрицательным, чтобы расчеты и будущие правила распределения платежей не теряли знак.

Для показаний электричества система сохраняет предупреждение, если предыдущее показание отсутствует или относится не к предыдущему месяцу. Предупреждение не блокирует внесение показания, но помогает оператору проверить пропущенный период до начислений.

В платежном разделе последние платежи, начисления, начисления поставщикам и показания счетчиков запрашиваются с явным серверным `limit=50` и показываются компактно. Если загружено больше строк, чем выводится на экран, под таблицей появляется счетчик `Показано 8 из N ...` с `aria-live="polite"`. Сводные итоги при этом считаются отдельным endpoint без экранного лимита. Последние начисления по гаражам и поставщикам открывают разбивку: объект учета, вид, источник, сумма, документ поставщика и комментарий. Dialog разбивки связан с месяцем учета через `aria-describedby`, поэтому заголовок и период начисления доступны вместе.

Для поступлений владельцев последние платежи показывают задолженность гаража до и после оплаты. Расчет строится от стартового баланса гаража, начислений до учетного месяца включительно и предыдущих поступлений по этому гаражу.

Для выплат поставщикам последние платежи показывают обязательство до и после выплаты. Расчет строится от стартового баланса поставщика, начислений поставщику до учетного месяца включительно и предыдущих выплат этому поставщику.

Финансовые записи отменяются без физического удаления: пользователь указывает причину, backend ставит `IsCanceled`, добавляет `Отменено: ...` в комментарий и пишет audit-событие. Поступления и выплаты используют `finance.operation_canceled`, начисления гаражей - `finance.accrual_canceled`, начисления поставщиков - `finance.supplier_accrual_canceled`, показания счетчиков - `finance.meter_reading_canceled`. Активные списки, сводка и расчеты задолженности исключают отмененные записи.

Audit создания ручных начислений показывает вид начисления, источник и комментарий; для начислений поставщикам дополнительно фиксируется номер документа или `без документа`. Это помогает проверять ручные корректировки без открытия каждой записи.

В справочнике гаражей есть единый поиск по номеру гаража и ФИО владельца. Поиск работает с кириллицей и обновляет список гаражей без перехода в отчеты.

Карточка гаража в справочнике открывает владельца, количество людей, этажи, стартовый баланс, стартовые показания воды/электричества и комментарий. Эти поля можно заполнить при создании гаража, чтобы после переноса из Access было видно исходное состояние без отдельного отчета. Dialog карточки связан с текущим владельцем через `aria-describedby`, поэтому заголовок гаража и контекст владельца доступны вместе.

Архивирование записей справочников и тарифов подтверждается отдельным dialog: первый клик только открывает проверку действия, а API вызывается после явного нажатия "Архивировать запись". Это защищает владельцев, гаражи, поставщиков, виды операций и тарифы от случайного архивирования, сохраняя soft-archive и audit-историю. Dialog архивирования связан с пояснением через `aria-describedby`, чтобы предупреждение о скрытии записи из рабочих списков было доступно вместе с заголовком.

База расчета тарифа проверяется на backend до сохранения: допустимы только `fixed`, `people`, `meter_water` и `meter_electricity`. Та же доменная константа используется при генерации регулярных начислений, поэтому ошибка настройки тарифа не откладывается до ежемесячного расчета.

Открытые dialog-окна закрываются клавишей `Escape`: это работает для карточки гаража, разбивки начисления и подтверждения архивирования. Клавиатурное закрытие не выполняет опасное действие, например не архивирует запись без отдельного подтверждения.

При открытии dialog интерфейс сразу переносит фокус внутрь окна: в карточке гаража и разбивке начисления фокус получает кнопка закрытия, а в подтверждении архивирования - безопасная кнопка "Отменить". Поэтому работа с клавиатуры начинается внутри активного окна, а случайное подтверждение архивирования через Enter не происходит.

Пока dialog открыт, клавиша `Tab` остается внутри него: в карточке гаража и разбивке начисления фокус не уходит на страницу под затемнением, а в подтверждении архивирования циклически проходит по закрытию, отмене и явному архивированию.

После закрытия dialog фокус возвращается на элемент, который его открыл, если этот элемент еще есть на странице. При настоящем архивировании запись исчезает из списка, поэтому фокус не возвращается в удаленную кнопку.

Клик по затемненному фону закрывает карточки и подтверждения так же безопасно, как `Escape`: опасные действия не выполняются, фокус возвращается на opener, а во время выполнения архивирования фон не закрывает confirmation dialog.

Списки справочников запрашиваются с явным серверным `limit=100`, показывают первые 5 записей и выводят счетчик вида `Показано 5 из N записей`, если в разделе есть дополнительные элементы. Так оператор видит, что список обрезан для компактности, а не считает данные потерянными. Кнопка `Показать все записи` раскрывает загруженный список внутри текущего справочника, а `Свернуть список` возвращает компактный режим; кнопка связана со списком через `aria-controls`, показывает состояние через `aria-expanded`, а счетчик обновляется в `aria-live="polite"` области.

При редактировании тарифа интерфейс предупреждает о несохраненных изменениях: если поля изменены и пользователь нажимает "Отменить" или пытается открыть другой тариф, система просит подтвердить сброс. Без подтверждения черновик остается в форме, после подтверждения форма возвращается в режим добавления или переключается на выбранный тариф. Тариф, который сейчас открыт в форме, помечается в списке как "Редактируется", а повторная кнопка открытия отключается, чтобы не сбросить поля случайным кликом.

Номер гаража уникален только среди активных записей: активный дубль запрещен сервисом и частичным уникальным индексом БД, но архивный гараж не блокирует повторное создание номера для актуальной записи или аккуратного переноса истории из Access.

Рабочие разделы frontend проверяют permissions до загрузки панелей: пользователи требуют `users.manage`, справочники - `dictionaries.read`, платежи - `payments.read`, отчеты - `reports.read`, импорт - `import.run`, audit - `audit.read`. Если права нет, система показывает понятное состояние "раздел недоступен" и не вызывает защищенный API этого раздела. Внутри доступных разделов действия записи также закрыты: справочники требуют `dictionaries.write`, тарифы - `tariffs.manage`, а платежи, начисления и показания счетчиков - `payments.write`.

Ошибки API возвращаются в едином формате `ProblemDetails`: `title` содержит код ошибки, `detail` - человекочитаемое описание, `status` - HTTP-статус, а `extensions.code` дублирует машинный код для frontend-обработки. Контракт применяется к auth, пользователям, справочникам, финансам, отчетам, импорту и "Что нового"; автоматические ошибки model validation используют код `validation_failed` и сохраняют `errors` по полям. Защищенные endpoints без входа отвечают кодом `unauthorized`, при недостатке прав - `forbidden`, а непредвиденные исключения - безопасным кодом `internal_error` без технических деталей в ответе.

На frontend ошибки форм и рабочих разделов выводятся через единый `FormError` с ролью `alert`. Это правило применяется к входу, смене пароля, пользователям, справочникам, платежам, импорту, audit, отчетам и "Что нового", чтобы backend-ошибки были заметны визуально и доступны для вспомогательных средств интерфейса.

Форма входа и создания первого администратора дополнительно показывает `validation summary` до обращения к API: проверяются email, имя администратора и парольная политика минимум 8 символов, заглавная буква, строчная буква и цифра. Если проверка не пройдена, API не вызывается, а пользователь видит список полей, которые нужно исправить. Та же клиентская проверка парольной политики применяется к форме смены пароля и созданию сотрудника: текущий пароль обязателен при смене, новый пароль должен соответствовать политике, повтор должен совпадать, а у нового сотрудника проверяются email, имя, пароль и роль. В справочниках validation summary добавлены для владельца, гаража, поставщика, группы поставщиков, видов поступлений/выплат и тарифа: пустые названия, короткий телефон, неверный ИНН, некорректный код и числовые значения останавливаются до отправки в API. В платежах такие же сводки есть для поступлений, выплат, ручных начислений, начислений поставщикам, регулярного месяца и показаний счетчиков. В отчетах проверяется порядок дат и месяцев для сводного отчета, поступлений, выплат и выгрузок.

Audit-журнал перед выдачей в API/UI маскирует чувствительные значения в описании события и идентификаторе сущности: email, Bearer-токены, значения рядом с `password`/`token`/`secret`/`api_key`/`ключ`, а также длинные номера. Это защищает журнал от случайного показа секретов, при этом обычные суммы и номера гаражей остаются читаемыми.

В интерфейсе audit-журнал запрашивает 50 последних событий текущего поиска, выводит первые 12 событий и показывает счетчик `Показано 12 из N событий` с `aria-live="polite"`, если найдено больше записей. CSV-экспорт при этом использует текущий поиск и не ограничивается экранным списком.

Audit-события тарифов показывают название, дату начала действия, базу расчета и ставку: это применяется к созданию, изменению и архивированию тарифа, чтобы администратор видел суть изменения прямо в журнале.

В разделе пользователей отображается матрица системных ролей и ключевых прав: управление пользователями, справочниками, тарифами, платежами, отчетами, импортом, audit и "Что нового". Матрица строится из `GET /api/users/roles`, поэтому показывает те же permissions, по которым backend и frontend закрывают действия.

Список пользователей в рабочей области запрашивается с явным серверным `limit=50` и выводится компактно: первые 8 сотрудников остаются на экране, а при более длинном списке под таблицей появляется счетчик `Показано 8 из N пользователей` с `aria-live="polite"`. Это делает ограничение списка явным для администратора и доступным для screen reader.

Любой авторизованный пользователь может сменить свой пароль в панели "Безопасность аккаунта". Frontend проверяет совпадение нового пароля и повтора до вызова API, а backend endpoint `PUT /api/auth/me/password` проверяет текущий пароль, запрещает оставить тот же пароль и пишет audit-события `auth.password_changed`/`auth.password_change_failed` без сохранения введенных паролей.

Пароль проверяется единым backend-правилом во всех точках: первый администратор, создание пользователя, админский сброс и самостоятельная смена пароля. Минимум: 8 символов, хотя бы одна заглавная буква, одна строчная буква и одна цифра; слабые пароли возвращают `password_policy_violation`.

JWT-секрет хранится только во внешней конфигурации: для локальной разработки можно использовать `Jwt:SigningKey` из `appsettings.json` или `JWT_SIGNING_KEY` в `.env`, но вне `Development` backend не запустится с примерным `change-this`/`development` ключом, пустым значением или ключом короче 32 UTF-8 байт. Для локального production-like запуска задавайте секрет через переменную окружения `Jwt__SigningKey`, user-secrets или секреты контура размещения.

Для будущих токенов 1C Fresh, фискального оборудования и других чувствительных интеграционных настроек добавлен сервис `ISensitiveDataProtector`: значения сохраняются только как строки с префиксом `gb:protected:v1:` и расшифровываются по отдельному purpose. Ключи ASP.NET Core Data Protection используют application name `GarageBalance`; для production-like стенда можно задать `DataProtection:KeysPath`, чтобы ключи переживали перезапуск контейнера или службы.

Зашифрованные секреты интеграций сохраняются в таблице `integration_secret_settings` через `IIntegrationSecretSettingsService`: уникальность задается парой provider/key без учета регистра, наружу возвращается только metadata, а plaintext доступен только сервисному коду через `GetSecretAsync`.

API-контракты дополнительно защищены backend-тестом от случайной выдачи секретов: response DTO не должны содержать поля вроде `password`, `passwordHash`, `signingKey`, `secret`, `apiKey` или `token`, кроме штатного `AuthResponse.AccessToken` в ответах входа и создания первого администратора.

API добавляет защитные HTTP-заголовки: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`. Для маршрутов `/api/*` дополнительно выставляется `Cache-Control: no-store, no-cache, max-age=0`, `Pragma: no-cache` и `Expires: 0`, чтобы ответы с персональными и финансовыми данными не сохранялись браузером или прокси.

Docker-nginx для frontend кэширует только хэшированные assets из `/assets/*` на 30 дней с `immutable`. `index.html` и SPA fallback отдаются с `Cache-Control: no-store` и отключенными `etag`/`if_modified_since`, чтобы телефон или прокси не оставались на старой HTML-оболочке после обновления стенда.

Frontend API-клиенты используют same-origin fallback: если `VITE_API_BASE_URL` не задан, запросы идут на текущий домен (`/api/...`) через nginx. Это важно для VPS и телефона: production-сборка не должна обращаться к `127.0.0.1:5080` на устройстве пользователя.

Статические файлы frontend включают `favicon.svg`, на который ссылается `index.html`. При размещении на VPS путь `/favicon.svg` должен отдавать SVG-файл, а не SPA fallback HTML.

Календарные даты платежей, начислений, счетчиков, тарифов и отчетов отображаются как `DateOnly` без UTC-преобразования: frontend вручную форматирует строки `YYYY-MM-DD` в `DD.MM.YYYY`, месяцы учета - в `MM.YYYY`, а дефолтные даты форм берет из локального календарного дня пользователя.

Месяц учета хранится как отдельный `DateOnly`-период первого дня месяца. Backend нормализует входящие месяцы через `MonthPeriod.Normalize`, поэтому дата `2026-06-23` в поле месяца учета сохраняется как `2026-06-01`, а frontend отображает ее как `06.2026`.

Источник начисления показывается в рабочих списках платежей: ручные корректировки помечаются как `Ручное`, регулярные начисления по тарифу - как `Авто`. Та же метка остается в разбивке начисления, чтобы бухгалтер видел происхождение суммы без сверки комментариев.

Регулярные начисления сохраняют сумму как отдельную историческую запись и дополняют комментарий снимком тарифа: название, ставка и дата начала действия. Если тариф позже изменить, уже созданные месяцы не пересчитываются и остаются с прежней суммой и прежним снимком ставки.

Audit запуска регулярных начислений фиксирует период, вид начисления, тариф, базу расчета, ставку, количество созданных строк, общую сумму и количество пропущенных гаражей. Это помогает сверять ежемесячный запуск без открытия каждой записи начисления.

Денежные суммы перед сохранением нормализуются на backend единым правилом: платежи, начисления и стартовые балансы округляются до 2 знаков через `MidpointRounding.AwayFromZero`, тарифные ставки - до 4 знаков. Это соответствует PostgreSQL precision `numeric(18,2)` для денег и `numeric(18,4)` для тарифов.

Стартовые и текущие показания счетчиков воды/электричества нормализуются до 3 знаков через `MidpointRounding.AwayFromZero`; расход считается из уже нормализованных значений и также сохраняется с точностью `numeric(18,3)`.

API Dockerfile собирает `net10.0` через `mcr.microsoft.com/dotnet/sdk:10.0`, публикует Release-сборку и запускает runtime-образ `mcr.microsoft.com/dotnet/aspnet:10.0` на `http://+:8080`. Этот контракт закреплен backend-тестом, чтобы Docker-упаковка не расходилась с целевой версией приложения.

`docker-compose.yml` поднимает сервисы в проверяемом порядке: PostgreSQL проходит `pg_isready`, API ждет здоровую БД и проверяет `/health` через `curl`, а frontend ждет здоровый API и проверяет nginx через `wget`. Это снижает риск, что локальный стенд покажет интерфейс раньше, чем backend реально готов принимать запросы.

## Правила проекта

Перед разработкой читать `AGENTS.md` и актуальный roadmap в `docs/`. Все пользовательские изменения должны отражаться в `backend/GarageBalance.Api/AppReleases/releases.json`, если они видны пользователю или меняют правила работы.

Приватные данные кооператива нельзя добавлять в Git: реальные `.env`, `appsettings.Local.json`, `.accdb`/`.mdb`, дампы, backup-файлы и папки `private-imports/`, `imports/private/`, `imports/raw/` должны оставаться только локально. Это правило закреплено в корневом `.gitignore` и backend-тесте `SensitiveFileGitIgnoreTests`.

Перед будущей публикацией в Git или упаковкой архива поставки запускайте `infrastructure/scripts/verify-package-privacy.ps1`: скрипт проверяет tracked/untracked Git-кандидаты и падает, если `.env`, `appsettings.Local.json`, Access-БД, дампы, backup или приватные import-папки видны Git.

Классы чувствительных данных, поля повышенной защиты, сроки хранения raw-импортов Access, quarantine/error bucket, логов и будущие требования к шифрованию описаны в `docs/security-data-protection.md`.

Главный roadmap: `docs/project-roadmap.md`.
