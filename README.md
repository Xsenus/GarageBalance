# GarageBalance

Веб-система финансового учета для гаражно-строительного кооператива.

Стартовый HTML приложения размечен как русскоязычный (`lang="ru"`), а вкладка браузера называется `GarageBalance - учет ГСК`, чтобы тестовый стенд и будущая production-сборка не выглядели как шаблонный Vite/frontend-проект.

## Стек

- Backend: C# / ASP.NET Core.
- Frontend: React + TypeScript + Vite.
- Database: PostgreSQL.
- Packaging: Docker Compose подготовлен заранее, финальная упаковка планируется ближе к сдаче.

## Структура

- `backend/GarageBalance.Api` - API, конфигурация, будущая бизнес-логика и `AppReleases`.
- `backend/GarageBalance.Api.Tests` - тесты backend.
- `frontend` - React-приложение.
- `docs` - roadmap, анализ исходных материалов, решения и история работ.
- `docker-compose.yml` - локальный запуск Postgres/API/frontend через Docker.

## Локальная разработка

Backend:

```powershell
dotnet tool restore
dotnet restore .\GarageBalance.slnx
dotnet tool run dotnet-ef database update --project .\backend\GarageBalance.Api\GarageBalance.Api.csproj --startup-project .\backend\GarageBalance.Api\GarageBalance.Api.csproj
dotnet run --project .\backend\GarageBalance.Api\GarageBalance.Api.csproj
```

Схема PostgreSQL меняется только через EF Core migrations в `backend/GarageBalance.Api/Infrastructure/Data/Migrations`. Production backend-код не должен вызывать `EnsureCreated` / `EnsureDeleted` или выполнять raw DDL вне папки миграций; это закреплено backend-тестом `DatabaseMigrationPolicyTests`, чтобы обновления чистой базы, импортной базы и VPS-стенда шли одним проверяемым путем.

Frontend:

```powershell
cd .\frontend
npm install
npm run dev
```

Для проверки production-сборки frontend:

```powershell
npm run build
npm run check:bundle
```

`npm run check:bundle` проверяет gzip-бюджет собранных JS/CSS assets: основной JS до 180 KiB, основной CSS до 40 KiB и общий JS/CSS budget до 260 KiB. Если добавляется тяжелая библиотека, превышение должно быть осознанным решением и фиксироваться в roadmap history.

Backend-списки и рабочие журналы должны ограничивать выборку до materialization: production-запросы к пользователям, справочникам, финансам, audit и import используют server-side `Take(...)` перед `ToListAsync(...)`. Это правило закреплено тестом `BackendPerformanceGuardTests`, а provider-specific fallback для SQLite остается явно ограниченным и не должен попадать в PostgreSQL-путь.

- `GET /api/import/access/runs/{id}/report` - скачивание JSON-отчета конкретного dry-run импорта.

Экран импорта Access показывает выбранный dry-run как наблюдаемую проверку: статус запуска, количество успешных проверок, предупреждений и ошибок, таблицу checks, пошаговый лог запуска и кнопку скачивания JSON-отчета. Пользовательские статусы выводятся по-русски: `Завершен`, `Заблокирован`, `Пройдено`, `Предупреждение`, `Ошибка`. История запусков показывает не только долю пройденных проверок, но и счетчики предупреждений/ошибок, например `2/3 · 1 предупреждение · 0 ошибок`. История, лог и карантин загружаются с явным серверным `limit`, чтобы панель импорта оставалась быстрой на длинной истории проверок и проблемных строк. Компактные таблицы лога, истории и карантина показывают счетчики `Показано X из Y`, если загружено больше строк, чем выводится на экран. Пустые состояния выбора запуска, проверок, лога, истории и карантина, а также счетчики компактных таблиц объявляются через `role="status"` и `aria-live="polite"`, чтобы после загрузки или закрытия последней проблемной строки было понятно, что список действительно пуст. Лог хранится в `access_import_run_log_entries`: UI получает безопасные шаги `file_received`, `hash_calculated`, `check_*`, `dry_run_finished`, уровень и сообщение, а служебный `DetailsJson` остается только в БД.

Имя выбранного Access-файла и строка текущего dry-run запуска тоже объявляются через `role="status"` и `aria-live="polite"`. Это помогает администратору сверить, какой файл поставлен на проверку и какой результат выбран в панели. Успешные сообщения панели импорта, включая готовность JSON-отчета и закрытие строки карантина, дополнительно размечаются как `role="status"`.

Для будущего фактического переноса Access добавлен реестр идемпотентности `access_import_row_fingerprints`: строка импорта регистрируется по источнику, типу сущности, внешнему id или SHA-256 row hash. Повторная регистрация возвращает существующий fingerprint и не должна создавать второй гараж, владельца, платеж или показание.

Проблемные строки будущего переноса Access складываются в `access_import_quarantine_items`: сервис хранит причину, уровень `error`/`warning`, SHA-256 row hash и JSON-снимок строки в БД, но публичные DTO показывают только безопасные поля без raw snapshot. В панели импорта администратор видит открытый карантин и может закрыть разобранную строку, действие фиксируется в audit.

- `GET /api/audit/events?limit=50` - audit-журнал действий пользователей и системных операций с серверным лимитом до 500 записей.
- `GET /api/audit/events/export` - CSV-экспорт текущей выборки audit-журнала с теми же фильтрами и маскированием чувствительных значений.
- `GET /api/reports/consolidated?limit=12` - консолидированный отчет за период с итогами по месяцам и гаражам; экран может передать `limit`, чтобы получить только видимые гаражные строки при сохранении полного `garageRowCount` и итогов.
- `GET /api/reports/consolidated/export/xlsx` - XLSX-выгрузка консолидированного отчета за период.
- `GET /api/reports/consolidated/export/pdf` - PDF-выгрузка консолидированного отчета за период.
- `GET /api/reports/income` - отчет по поступлениям: начисления, оплаты, фильтры по датам, поиску, гаражам, владельцам и видам поступлений; экран может передать `limit`, чтобы получить только видимые строки при сохранении полного `rowCount` и итогов.
- `GET /api/reports/income/export/xlsx` - XLSX-выгрузка отчета по поступлениям с теми же фильтрами, что экранный отчет.
- `GET /api/reports/income/export/pdf` - PDF-выгрузка отчета по поступлениям с теми же фильтрами, что экранный отчет.
- `GET /api/reports/expense` - отчет по выплатам: поставщики, виды выплат, документы, период и режим строк; экран может передать `limit`, чтобы получить только видимые строки при сохранении полного `rowCount` и итогов.
- `GET /api/reports/expense/export/xlsx` - XLSX-выгрузка отчета по выплатам с теми же фильтрами, что экранный отчет.
- `GET /api/reports/expense/export/pdf` - PDF-выгрузка отчета по выплатам с теми же фильтрами, что экранный отчет.
- `GET /api/app-releases` - история обновлений для раздела "Что нового" после входа в систему.

Во время запроса раздел "Что нового" объявляет статус `Загружаем историю обновлений...` через `role="status"` и `aria-live="polite"`. Если история обновлений пуста, раздел показывает `Пока нет опубликованных изменений.` и тоже объявляет это состояние через `role="status"` и `aria-live="polite"`, чтобы пользователь отличал пустой список от загрузки или ошибки.

При первом применении миграций чистая PostgreSQL-БД получает системные виды поступлений и выплат из ТЗ. Виды поступлений: вода, мусор, электроэнергия, членский, целевой и вступительный взносы, подключения, штраф и предписание. Виды выплат: электроэнергия, вывоз мусора, водоснабжение, банковские, юридические, зарплата, прочие расходы и штрафы. Эти записи создаются с `IsSystem=true`, не удаляются как обычные пользовательские значения и не дублируются, если название уже было заведено вручную.

Длинные экранные отчеты показывают, что таблица обрезана до рабочего лимита: для гаражей выводится `Показано 12 из N строк`, для поступлений и выплат - `Показано 16 из N строк`. Экранные запросы сводного отчета, поступлений и выплат передают backend `limit`, но итоговые суммы и счетчики `garageRowCount`/`rowCount` считаются по полной выборке. Эти счетчики размечены `role="status"` и `aria-live="polite"`, чтобы изменение количества видимых строк было доступно screen reader без ручного поиска по таблице.

Если фильтр сводного отчета, отчета по поступлениям или отчета по выплатам не нашел строк, таблица оставляет заголовки и показывает понятное пустое состояние с `role="status"` и `aria-live="polite"`. В сводном отчете это правило применяется и к помесячной таблице, и к строкам гаражей. Пользователь сразу видит, что отчет сформирован корректно, но данных по выбранным условиям нет.

На узкой ширине рабочая область перестраивается в одну колонку, строки таблиц становятся вертикальными, а основная навигация превращается в горизонтально прокручиваемую полосу. Для длинных значений в строках таблиц задано `min-width: 0`, чтобы текст переносился внутри ячейки и не растягивал экран телефона.

Экран отчетов сохраняет последние примененные фильтры сводного отчета, поступлений и выплат в `sessionStorage` текущей вкладки. При повторном открытии рабочей области первый запрос отчетов использует эти значения; испорченные или устаревшие данные sessionStorage игнорируются и заменяются дефолтами текущего периода.

В справочниках есть быстрый поиск гаражей по номеру или ФИО владельца и поиск поставщиков по названию, ИНН или контактному лицу. Оба поиска обращаются к backend-фильтрам, автоматически запускаются после короткой паузы при вводе и показывают счетчик найденных записей; кнопка рядом с полем остается для ручного запуска. Если очистить поле поиска, интерфейс автоматически возвращает полный список гаражей или поставщиков и показывает статус сброса фильтра. Статусы `Найдено гаражей`, `Найдено поставщиков`, `Показаны все гаражи` и `Показаны все поставщики` объявляются через `role="status"` и `aria-live="polite"`.

Поля быстрого поиска закреплены индексами в БД: гаражи индексируются по номеру и владельцу, владельцы - по ФИО и телефону, поставщики - по названию, группе, ИНН и контактному лицу. Для PostgreSQL дополнительно применяется `pg_trgm` и GIN trigram-индексы по активным записям владельцев, гаражей и поставщиков, чтобы поиск по подстроке оставался быстрым после переноса данных из Access.

Стартовый баланс гаража учитывается как начальная задолженность в отчетах по поступлениям и сводном отчете; стартовый баланс поставщика учитывается как начальное обязательство в отчете по выплатам. Для гаражей добавлена migration `GarageStartingBalance`.

Если расчет задолженности гаража уходит ниже нуля, frontend показывает это как "Переплата" и выводит сумму без минуса. Исходное поле API `debt` остается отрицательным, чтобы расчеты и будущие правила распределения платежей не теряли знак.

Для показаний электричества система сохраняет предупреждение, если предыдущее показание отсутствует или относится не к предыдущему месяцу. Предупреждение не блокирует внесение показания, но помогает оператору проверить пропущенный период до начислений.

В разделе "Счетчики" система дополнительно показывает гаражи, по которым за выбранный месяц еще нет активного показания воды или электричества. Предупреждение строится по активным гаражам, учитывает отмененные показания как отсутствующие и не показывает архивные гаражи; строка объявляется через `role="status"` и `aria-live="polite"`.

Audit создания показаний счетчиков фиксирует гараж, тип счетчика, учетный месяц, дату, предыдущее и текущее значения, расход, наличие предупреждения и комментарий. Это помогает сверить воду/электричество перед регулярными начислениями.

Если показание счетчика отменено и затем внесено заново за тот же месяц, регулярное начисление по воде или электричеству использует только активное повторное показание. Отмененная строка остается в audit и истории отмен, но не участвует в сумме начисления; уникальный индекс БД также ограничивает только активные показания, чтобы повторный ввод после отмены не блокировался.

В платежном разделе последние платежи, начисления, начисления поставщикам и показания счетчиков запрашиваются с явным серверным `limit=50` и показываются компактно. Если загружено больше строк, чем выводится на экран, под таблицей появляется счетчик `Показано 8 из N ...` с `role="status"` и `aria-live="polite"`. Если после загрузки или отмены строк список стал пустым, сообщение `Операций пока нет`, `Начислений пока нет`, `Начислений поставщикам пока нет` или `Показаний пока нет` тоже объявляется через `role="status"` и `aria-live="polite"`. Сводные итоги при этом считаются отдельным endpoint без экранного лимита. Последние начисления по гаражам и поставщикам открывают разбивку: объект учета, вид, источник, сумма, документ поставщика и комментарий. Dialog разбивки связан с месяцем учета через `aria-describedby`, поэтому заголовок и период начисления доступны вместе.

Для рабочих таблиц платежей используется серверная пагинация: frontend передает `offset`, `limit`, период, тип и поиск, а backend возвращает `items`, `totalCount`, `offset` и `limit` без загрузки полного списка в память приложения. Это правило закреплено backend-тестами и performance guard для операций, начислений владельцам, начислений поставщикам и показаний счетчиков; сводные карточки периода обновляются отдельным summary-запросом и не зависят от размера текущей страницы.

В поступлениях владельцев и выплатах поставщикам есть отдельное поле `Комментарий платежа`. Оно сохраняется в `FinancialOperation.Comment`, показывается отдельной колонкой в рабочих таблицах приходов/расходов, попадает в audit и участвует в серверном поиске по платежам. Комментарии начислений за месяц остаются в строках начислений, поэтому бухгалтер может отдельно фиксировать пояснение к долгу месяца и пояснение к фактической оплате.

В форме поступления есть отдельный поиск гаража по номеру или ФИО владельца. Поиск обращается к backend `GET /api/dictionaries/garages?search=...&limit=100`, обновляет только варианты выбора для поступления и показывает статус `Найдено гаражей: N`, поэтому оператор может провести оплату по гаражу, которого нет в первой загруженной странице справочника.

Для поступлений владельцев последние платежи показывают задолженность гаража до и после оплаты. Расчет строится от стартового баланса гаража, начислений до учетного месяца включительно и предыдущих поступлений по этому гаражу.

Для выплат поставщикам последние платежи показывают обязательство до и после выплаты. Расчет строится от стартового баланса поставщика, начислений поставщику до учетного месяца включительно и предыдущих выплат этому поставщику.

Распределение поступлений владельцев и выплат поставщикам зафиксировано как автоматическое закрытие старейшего долга. Сначала учитываются уже проведенные предыдущие оплаты и стартовая переплата, затем текущий платеж закрывает стартовый долг, самый ранний месяц начисления и следующие месяцы по порядку. В ответе операции поле `paymentAllocations` показывает тип строки (`starting_balance`, `month`, `overpayment`), месяц, долг до строки, сумму закрытия и остаток после строки; frontend выводит эту разбивку рядом с долгом до/после.

Финансовые записи отменяются без физического удаления: пользователь указывает причину, backend ставит `IsCanceled`, добавляет `Отменено: ...` в комментарий и пишет audit-событие. Поступления и выплаты используют `finance.operation_canceled`, начисления гаражей - `finance.accrual_canceled`, начисления поставщиков - `finance.supplier_accrual_canceled`, показания счетчиков - `finance.meter_reading_canceled`. Активные списки, сводка и расчеты задолженности исключают отмененные записи.

Поступления и выплаты можно исправить прямым редактированием строки в платежной таблице. Backend пересчитывает долг/обязательство по обновленным данным, проверяет дубли документа и пишет `finance.income_updated` или `finance.expense_updated` с audit-summary формата `было ...; стало ...`, чтобы проверяющий видел старую и новую сумму, дату, месяц, контрагента, вид операции, документ и комментарий платежа.

После отмены ошибочного поступления или выплаты можно внести корректную замену с тем же номером документа и датой. Уникальный индекс платежных документов ограничивает только активные записи с заполненным номером документа, поэтому отмененная строка остается в audit, но не блокирует исправление.

После отмены ошибочного начисления можно внести замену за тот же гараж/вид/месяц/источник, а после отмены начисления поставщику - замену за того же поставщика/вид/месяц/источник/документ. Уникальные индексы БД ограничивают только активные строки, поэтому история отмен сохраняется, но не блокирует исправление.

Audit отмены финансовых записей показывает не только причину, но и проверочный контекст: для платежей дату, учетный месяц, вид и документ; для начислений вид, источник и документ поставщика; для показаний месяц, дату и расход.

Audit создания поступлений и выплат показывает сумму, контрагента, дату операции, учетный месяц, вид операции, документ или `без документа` и комментарий. Поэтому сверка платежей в журнале не требует открывать каждую карточку операции.

Audit создания ручных начислений показывает вид начисления, источник и комментарий; для начислений поставщикам дополнительно фиксируется номер документа или `без документа`. Это помогает проверять ручные корректировки без открытия каждой записи.

Ручные начисления поставщикам требуют комментарий так же, как ручные начисления по гаражам. Если комментарий пустой, backend отклоняет запись до сохранения и не пишет audit, чтобы в обязательствах поставщика не появлялись необъясненные корректировки.

В справочнике гаражей есть единый поиск по номеру гаража и ФИО владельца. Поиск работает с кириллицей и обновляет список гаражей без перехода в отчеты.

Карточка гаража в справочнике открывает владельца, количество людей, этажи, стартовый баланс, стартовые показания воды/электричества и комментарий. В табличном справочнике гаражей через ПКМ доступна "История баланса": backend строит ее по стартовому балансу, начислениям и поступлениям за выбранный период, а frontend показывает итоги и помесячную таблицу задолженности. Эти поля можно заполнить при создании гаража, чтобы после переноса из Access было видно исходное состояние без отдельного отчета. Dialog карточки связан с текущим владельцем через `aria-describedby`, поэтому заголовок гаража и контекст владельца доступны вместе.

Архивирование записей справочников и тарифов подтверждается отдельным dialog: первый клик только открывает проверку действия, а API вызывается после явного нажатия "Архивировать запись". Это защищает владельцев, гаражи, поставщиков, виды операций и тарифы от случайного архивирования, сохраняя soft-archive и audit-историю. Dialog архивирования связан с пояснением через `aria-describedby`, чтобы предупреждение о скрытии записи из рабочих списков было доступно вместе с заголовком.

База расчета тарифа проверяется на backend до сохранения: допустимы только `fixed`, `people`, `meter_water` и `meter_electricity`. Та же доменная константа используется при генерации регулярных начислений, поэтому ошибка настройки тарифа не откладывается до ежемесячного расчета.

Для тарифа `meter_electricity` можно оставить обычную единую ставку или заполнить трехзонную схему: первый порог, второй порог, ставку до первого порога, ставку между порогами и ставку сверх второго порога. Если заполнена только часть полей, backend отклоняет тариф; если схема заполнена полностью, регулярное начисление умножает расход счетчика по зонам и сохраняет снимок порогов/ставок в комментарии начисления и audit.

Регулярные начисления проверяют совместимость системного вида поступления с базой тарифа. Вода принимает только `meter_water`, мусор - `people`, электроэнергия - `meter_electricity`, а членский, целевой, вступительный взнос и подключения - только `fixed`. Frontend фильтрует список тарифов по выбранному виду начисления, а backend повторно проверяет пару перед созданием строк.

Зарплата персонала начисляется из раздела платежей по группе поставщиков/сотрудников. Команда "Зарплата группы" создает регулярные начисления поставщикам с системным видом выплаты `salary` для каждого активного участника выбранной группы; повторный запуск за тот же месяц и документ пропускает уже созданные строки, а сами начисления дальше редактируются и отменяются через общую таблицу начислений поставщикам.

Открытые dialog-окна закрываются клавишей `Escape`: это работает для карточки гаража, разбивки начисления и подтверждения архивирования. Клавиатурное закрытие не выполняет опасное действие, например не архивирует запись без отдельного подтверждения.

Все рабочие dialog-окна на базе `detail-dialog` должны иметь `role="dialog"`, `aria-modal="true"`, доступное имя через `aria-labelledby` и описание через `aria-describedby`. Это правило закреплено frontend structural test, чтобы будущие модальные окна не появлялись без базовой доступной разметки.

При открытии dialog интерфейс сразу переносит фокус внутрь окна: в карточке гаража и разбивке начисления фокус получает кнопка закрытия, а в подтверждении архивирования - безопасная кнопка "Отменить". Поэтому работа с клавиатуры начинается внутри активного окна, а случайное подтверждение архивирования через Enter не происходит.

Пока dialog открыт, клавиша `Tab` остается внутри него: в карточке гаража и разбивке начисления фокус не уходит на страницу под затемнением, а в подтверждении архивирования циклически проходит по закрытию, отмене и явному архивированию.

После закрытия dialog фокус возвращается на элемент, который его открыл, если этот элемент еще есть на странице. При настоящем архивировании запись исчезает из списка, поэтому фокус не возвращается в удаленную кнопку.

Клик по затемненному фону закрывает карточки и подтверждения так же безопасно, как `Escape`: опасные действия не выполняются, фокус возвращается на opener, а во время выполнения архивирования фон не закрывает confirmation dialog.

Списки справочников запрашиваются с явным серверным `limit=100`, показывают первые 5 записей и выводят счетчик вида `Показано 5 из N записей`, если в разделе есть дополнительные элементы. Так оператор видит, что список обрезан для компактности, а не считает данные потерянными. Кнопка `Показать все записи` раскрывает загруженный список внутри текущего справочника, а `Свернуть список` возвращает компактный режим; кнопка связана со списком через `aria-controls`, показывает состояние через `aria-expanded`, а этот disclosure-контракт закреплен frontend structural test. Счетчик обновляется в `role="status"` и `aria-live="polite"` области. Если конкретный справочник пуст, общий список объявляет сообщение вроде `Владельцев пока нет` через `role="status"` и `aria-live="polite"`.

При редактировании тарифа интерфейс предупреждает о несохраненных изменениях: если поля изменены и пользователь нажимает "Отменить" или пытается открыть другой тариф, система просит подтвердить сброс. Без подтверждения черновик остается в форме, после подтверждения форма возвращается в режим добавления или переключается на выбранный тариф. Тариф, который сейчас открыт в форме, помечается в списке как "Редактируется", а повторная кнопка открытия отключается, чтобы не сбросить поля случайным кликом. Предупреждение `Есть несохраненные изменения тарифа.` объявляется через `role="status"` и `aria-live="polite"`.

Номер гаража уникален только среди активных записей: активный дубль запрещен сервисом и частичным уникальным индексом БД, но архивный гараж не блокирует повторное создание номера для актуальной записи или аккуратного переноса истории из Access.

Рабочие разделы frontend проверяют permissions до загрузки панелей: пользователи требуют `users.manage`, справочники - `dictionaries.read`, платежи - `payments.read`, отчеты - `reports.read`, импорт - `import.run`, audit - `audit.read`. Если права нет, система показывает понятное состояние "раздел недоступен" и не вызывает защищенный API этого раздела. Внутри доступных разделов действия записи также закрыты: справочники требуют `dictionaries.write`, тарифы - `tariffs.manage`, а платежи, начисления и показания счетчиков - `payments.write`.

Боковое меню рабочей области переключает настоящий активный раздел: одновременно рендерится только выбранная панель, пункт `Тарифы` открывает рабочий блок справочников с тарифами, а предыдущие разделы уходят со страницы и не накладываются друг на друга.

Ошибки API возвращаются в едином формате `ProblemDetails`: `title` содержит код ошибки, `detail` - человекочитаемое описание, `status` - HTTP-статус, а `extensions.code` дублирует машинный код для frontend-обработки. Контракт применяется к auth, пользователям, справочникам, финансам, отчетам, импорту и "Что нового"; автоматические ошибки model validation используют код `validation_failed` и сохраняют `errors` по полям. Защищенные endpoints без входа отвечают кодом `unauthorized`, при недостатке прав - `forbidden`, а непредвиденные исключения - безопасным кодом `internal_error` без технических деталей в ответе.

На frontend ошибки форм и рабочих разделов выводятся через единый `FormError` с ролью `alert`. Это правило применяется к входу, смене пароля, пользователям, справочникам, платежам, импорту, audit, отчетам и "Что нового", чтобы backend-ошибки были заметны визуально и доступны для вспомогательных средств интерфейса. Общая разметка `FormError` и `FormValidationSummary` закреплена frontend structural test, поэтому новые правки не должны убирать alert-семантику у ошибок и сводок валидации.

Форма входа и создания первого администратора дополнительно показывает `validation summary` до обращения к API: проверяются email, имя администратора и парольная политика минимум 8 символов, заглавная буква, строчная буква и цифра. Если проверка не пройдена, API не вызывается, а пользователь видит список полей, которые нужно исправить. Та же клиентская проверка парольной политики применяется к форме смены пароля и созданию сотрудника: текущий пароль обязателен при смене, новый пароль должен соответствовать политике, повтор должен совпадать, а у нового сотрудника проверяются email, имя, пароль и роль. В справочниках validation summary добавлены для владельца, гаража, поставщика, группы поставщиков, видов поступлений/выплат и тарифа: пустые названия, короткий телефон, неверный ИНН, некорректный код и числовые значения останавливаются до отправки в API. В платежах такие же сводки есть для поступлений, выплат, ручных начислений, начислений поставщикам, регулярного месяца и показаний счетчиков. В отчетах проверяется порядок дат и месяцев для сводного отчета, поступлений, выплат и выгрузок.

Audit-журнал перед выдачей в API/UI маскирует чувствительные значения в описании события и идентификаторе сущности: email, Bearer-токены, значения рядом с `password`/`token`/`secret`/`api_key`/`ключ`, а также длинные номера. Это защищает журнал от случайного показа секретов, при этом обычные суммы и номера гаражей остаются читаемыми.

В интерфейсе audit-журнал запрашивает 50 последних событий текущего поиска, выводит первые 12 событий и показывает счетчик `Показано 12 из N событий` с `role="status"` и `aria-live="polite"`, если найдено больше записей. Если событий нет, таблица остается на месте и объявляет сообщение `Событий пока нет` через `role="status"` и `aria-live="polite"`. CSV-экспорт при этом использует текущий поиск и не ограничивается экранным списком.

Успешные сообщения выгрузок в импорте, audit и отчетах объявляются через `aria-live="polite"`: `Отчет dry-run импорта готов.`, `Audit-журнал CSV готов.`, а также XLSX/PDF-статусы сводного отчета, поступлений и выплат. Сообщения импорта, CSV-выгрузки audit и XLSX/PDF-выгрузок отчетов дополнительно размечаются как `role="status"`, чтобы пользователь получал подтверждение готовности файла без ручного поиска нового текста на странице.

Все polite live-region сообщения основного рабочего интерфейса должны одновременно иметь `role="status"`. Это правило закреплено frontend structural test, чтобы новые динамические статусы не появлялись без доступной семантики.

Иконкнопки без видимого текста должны оставаться настоящими `<button>` с явным `type="button"` или `type="submit"` и доступным именем через `aria-label`. Это закреплено frontend structural test, чтобы действия вроде выхода, поиска, закрытия dialog, открытия карточки и архивирования не теряли клавиатурную и screen reader-доступность.

Все `<button>` в основном React-приложении должны иметь явный `type="button"` или `type="submit"`. Это закреплено frontend structural test, чтобы новые кнопки внутри форм не получали браузерный submit по умолчанию.

Все поля форм (`input`, `select`, `textarea`) должны иметь явное доступное имя через `aria-label` или `aria-labelledby`. Стартовая форма входа и первого администратора, платежи, справочники, отчеты, импорт и audit закреплены frontend structural test, чтобы новые поля не появлялись без понятного имени для screen reader.

Подсказки парольной политики в формах входа/первого администратора, смены пароля и создания пользователя связаны с password-полями через `aria-describedby`. Это закреплено frontend structural test, чтобы правило "минимум 8 символов, заглавная буква, строчная буква и цифра" было доступно вместе с самим полем.

Audit-события тарифов показывают название, дату начала действия, базу расчета и ставку: это применяется к созданию, изменению и архивированию тарифа, чтобы администратор видел суть изменения прямо в журнале.

В разделе пользователей отображается матрица системных ролей и ключевых прав: управление пользователями, справочниками, тарифами, платежами, отчетами, импортом, audit и "Что нового". Матрица строится из `GET /api/users/roles`, поэтому показывает те же permissions, по которым backend и frontend закрывают действия.

Список пользователей в рабочей области запрашивается с явным серверным `limit=50` и выводится компактно: первые 8 сотрудников остаются на экране, а при более длинном списке под таблицей появляется счетчик `Показано 8 из N пользователей` с `role="status"` и `aria-live="polite"`. Если пользователей или ролей еще нет, таблицы объявляют `Пользователей пока нет` и `Роли пока не загружены` через `role="status"` и `aria-live="polite"`, чтобы администратор отличал пустой список от незавершенной загрузки.

Любой авторизованный пользователь может сменить свой пароль в панели "Безопасность аккаунта". Frontend проверяет совпадение нового пароля и повтора до вызова API, а backend endpoint `PUT /api/auth/me/password` проверяет текущий пароль, запрещает оставить тот же пароль и пишет audit-события `auth.password_changed`/`auth.password_change_failed` без сохранения введенных паролей. Успешное подтверждение смены пароля объявляется через `role="status"` и `aria-live="polite"`, чтобы пользователь сразу понимал, что сохранение завершилось.

Пароль проверяется единым backend-правилом во всех точках: первый администратор, создание пользователя, админский сброс и самостоятельная смена пароля. Минимум: 8 символов, хотя бы одна заглавная буква, одна строчная буква и одна цифра; слабые пароли возвращают `password_policy_violation`.

JWT-секрет хранится только во внешней конфигурации: для локальной разработки можно использовать `Jwt:SigningKey` из `appsettings.json` или `JWT_SIGNING_KEY` в `.env`, но вне `Development` backend не запустится с примерным `change-this`/`development` ключом, пустым значением или ключом короче 32 UTF-8 байт. Для локального production-like запуска задавайте секрет через переменную окружения `Jwt__SigningKey`, user-secrets или секреты контура размещения.

Для будущих токенов 1C Fresh, фискального оборудования и других чувствительных интеграционных настроек добавлен сервис `ISensitiveDataProtector`: значения сохраняются только как строки с префиксом `gb:protected:v1:` и расшифровываются по отдельному purpose. Ключи ASP.NET Core Data Protection используют application name `GarageBalance`; для production-like стенда можно задать `DataProtection:KeysPath`, чтобы ключи переживали перезапуск контейнера или службы.

Зашифрованные секреты интеграций сохраняются в таблице `integration_secret_settings` через `IIntegrationSecretSettingsService`: уникальность задается парой provider/key без учета регистра, наружу возвращается только metadata, а plaintext доступен только сервисному коду через `GetSecretAsync`.

API-контракты дополнительно защищены backend-тестом от случайной выдачи секретов: response DTO не должны содержать поля вроде `password`, `passwordHash`, `signingKey`, `secret`, `apiKey` или `token`, кроме штатного `AuthResponse.AccessToken` в ответах входа и создания первого администратора.

API добавляет защитные HTTP-заголовки: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`. Для маршрутов `/api/*` дополнительно выставляется `Cache-Control: no-store, no-cache, max-age=0`, `Pragma: no-cache` и `Expires: 0`, чтобы ответы с персональными и финансовыми данными не сохранялись браузером или прокси.

Docker-nginx для frontend кэширует только хэшированные assets из `/assets/*` на 30 дней с `immutable`. `index.html` и SPA fallback отдаются с `Cache-Control: no-store` и отключенными `etag`/`if_modified_since`, чтобы телефон или прокси не оставались на старой HTML-оболочке после обновления стенда.

Frontend API-клиенты используют same-origin fallback: если `VITE_API_BASE_URL` не задан, запросы идут на текущий домен (`/api/...`) через nginx. Это важно для VPS и телефона: production-сборка не должна обращаться к `127.0.0.1:5080` на устройстве пользователя.

Статические файлы frontend включают `favicon.svg`, на который ссылается `index.html`. При размещении на VPS путь `/favicon.svg` должен отдавать SVG-файл, а не SPA fallback HTML.

Календарные даты платежей, начислений, счетчиков, тарифов и отчетов отображаются как `DateOnly` без UTC-преобразования: frontend вручную форматирует строки `YYYY-MM-DD` в `DD.MM.YYYY`, месяцы учета - в `MM.YYYY`, а дефолтные даты форм берет из локального календарного дня пользователя.

Месяц учета хранится как отдельный `DateOnly`-период первого дня месяца. Backend нормализует входящие месяцы через `MonthPeriod.Normalize`, поэтому дата `2026-06-23` в поле месяца учета сохраняется как `2026-06-01`, а frontend отображает ее как `06.2026`.

Источник начисления показывается в рабочих списках платежей: ручные корректировки помечаются как `Ручное`, регулярные начисления по тарифу - как `Авто`. Та же метка остается в разбивке начисления, чтобы бухгалтер видел происхождение суммы без сверки комментариев.

Регулярные начисления сохраняют сумму как отдельную историческую запись и дополняют комментарий снимком тарифа: название, ставка и дата начала действия. Если тариф позже изменить, уже созданные месяцы не пересчитываются и остаются с прежней суммой и прежним снимком ставки. После запуска формы итог `Создано N, пропущено M` объявляется через `role="status"` и `aria-live="polite"`, чтобы результат ежемесячного расчета был доступен без ручного поиска по списку начислений.

Новые регулярные начисления дополнительно сохраняют ссылку на использованный тариф. Если по тарифу уже есть активное регулярное начисление, дату начала действия нельзя перенести позже самого раннего начисленного месяца; для новой ставки нужно создать новую версию тарифа, а старые суммы и снимки остаются без пересчета.

Audit запуска регулярных начислений фиксирует период, вид начисления, тариф, базу расчета, ставку, количество созданных строк, общую сумму и количество пропущенных гаражей. Это помогает сверять ежемесячный запуск без открытия каждой записи начисления.

Денежные суммы перед сохранением нормализуются на backend единым правилом: платежи, начисления и стартовые балансы округляются до 2 знаков через `MidpointRounding.AwayFromZero`, тарифные ставки - до 4 знаков. Это соответствует PostgreSQL precision `numeric(18,2)` для денег и `numeric(18,4)` для тарифов.

Стартовые и текущие показания счетчиков воды/электричества нормализуются до 3 знаков через `MidpointRounding.AwayFromZero`; расход считается из уже нормализованных значений и также сохраняется с точностью `numeric(18,3)`.

API Dockerfile собирает `net10.0` через `mcr.microsoft.com/dotnet/sdk:10.0`, публикует Release-сборку и запускает runtime-образ `mcr.microsoft.com/dotnet/aspnet:10.0` на `http://+:8080`. Этот контракт закреплен backend-тестом, чтобы Docker-упаковка не расходилась с целевой версией приложения.

`docker-compose.yml` поднимает сервисы в проверяемом порядке: PostgreSQL проходит `pg_isready`, API ждет здоровую БД и проверяет `/health` через `curl`, а frontend ждет здоровый API и проверяет nginx через `wget`. Это снижает риск, что локальный стенд покажет интерфейс раньше, чем backend реально готов принимать запросы.

## Правила проекта

Перед разработкой читать `AGENTS.md` и актуальный roadmap в `docs/`. Все пользовательские изменения должны отражаться в `backend/GarageBalance.Api/AppReleases/releases.json`, если они видны пользователю или меняют правила работы.

Контроллеры остаются тонким HTTP-слоем: принимают DTO, проверяют авторизацию, вызывают application-сервисы и возвращают HTTP-ответ. Прямые зависимости контроллеров от EF Core, `GarageBalanceDbContext`, репозиториев, `Infrastructure.Data`, raw SQL/DDL и возврат Domain-сущностей через action result запрещены и закреплены backend-тестом `ControllerThinnessTests`.

Приватные данные кооператива нельзя добавлять в Git: реальные `.env`, `appsettings.Local.json`, `.accdb`/`.mdb`, дампы, backup-файлы и папки `private-imports/`, `imports/private/`, `imports/raw/` должны оставаться только локально. Это правило закреплено в корневом `.gitignore` и backend-тесте `SensitiveFileGitIgnoreTests`.

Перед будущей публикацией в Git или упаковкой архива поставки запускайте `infrastructure/scripts/verify-package-privacy.ps1`: скрипт проверяет tracked/untracked Git-кандидаты и падает, если `.env`, `appsettings.Local.json`, Access-БД, дампы, backup или приватные import-папки видны Git.

Классы чувствительных данных, поля повышенной защиты, сроки хранения raw-импортов Access, quarantine/error bucket, логов и будущие требования к шифрованию описаны в `docs/security-data-protection.md`.

Главный roadmap: `docs/project-roadmap.md`.
