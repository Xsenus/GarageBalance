# Реестр дополнительных отчетов

Документ резервирует до трех дополнительных отчетов после базовых отчетов: консолидированного отчета, отчета по поступлениям и отчета по выплатам. Реестр нужен, чтобы новые отчеты не появлялись случайно и проходили единый путь из `docs/new-report-checklist.md`.

## Общие Правила

1. Каждый новый отчет должен отвечать на один понятный бизнес-вопрос.
2. Каждый отчет должен быть закрыт правом `reports.read`.
3. Для каждого отчета заранее фиксируются период, источник данных, база учета сумм, фильтры, экспорт и acceptance-критерии.
4. Если отчет участвует в закрытии месяца, он добавляется в `docs/monthly-cycle-checklist.md`.
5. Если отчет виден пользователю, добавляется запись в `backend/GarageBalance.Api/AppReleases/releases.json`.
6. Для каждого отчета нужны backend tests, React tests, проверка XLSX/PDF при наличии экспорта и запись в историю `docs/project-roadmap.md`.

## Слот 1. Задолженность И Переплаты

Бизнес-вопрос: какие гаражи имеют задолженность или переплату на конец выбранного периода.

Планируемые фильтры:

- период `ММ.ГГГГ`;
- гаражи;
- владельцы;
- только задолженность;
- только переплата;
- минимальная сумма отклонения.

Данные:

- гаражи и владельцы;
- стартовые балансы;
- начисления по учетному месяцу;
- поступления по фактической дате оплаты;
- отмененные операции исключаются.

Ожидаемые итоги:

- начислено;
- оплачено;
- задолженность;
- переплата;
- количество гаражей с долгом;
- количество гаражей с переплатой.

Экспорт: XLSX и PDF.

Acceptance:

- отчет объясняет расхождение между начислено и оплачено;
- строки по гаражам совпадают с карточкой гаража и сводным отчетом;
- выгрузка использует те же фильтры, что экран.

## Слот 2. Счетчики И Расход

Бизнес-вопрос: какие гаражи дали расход воды/электроэнергии за период и где есть пропуски или резкие отклонения.

Планируемые фильтры:

- период `ММ.ГГГГ`;
- тип счетчика: `meter_water`, `meter_electricity`;
- гаражи;
- владельцы;
- только пропущенные показания;
- только отрицательный или подозрительный расход.

Данные:

- показания счетчиков;
- гаражи и владельцы;
- тарифы по счетчикам;
- регулярные начисления по `meter_water` и `meter_electricity`.

Ожидаемые итоги:

- начальные показания;
- конечные показания;
- расход;
- начислено по расходу;
- пропущенные показания;
- подозрительные строки.

Экспорт: XLSX; PDF по решению заказчика после первого рабочего месяца.

Acceptance:

- отчет помогает найти гаражи без показаний до генерации начислений;
- начисленная сумма совпадает с регулярными начислениями по счетчикам;
- предупреждения не блокируют отчет, но явно показывают проблемные строки.

## Слот 3. Поставщики И Обязательства

Бизнес-вопрос: какие обязательства перед поставщиками начислены, оплачены и остаются открытыми за период.

Планируемые фильтры:

- период `ДД.ММ.ГГГГ`;
- поставщики;
- группы поставщиков;
- виды выплат;
- режим строк: `all`, `accruals`, `payments`;
- только неоплаченные обязательства.

Данные:

- начисления поставщикам по учетному месяцу;
- фактические выплаты по дате оплаты;
- поставщики и группы поставщиков;
- виды выплат;
- отмененные начисления и выплаты исключаются.

Ожидаемые итоги:

- начислено поставщикам;
- выплачено;
- разница;
- количество открытых обязательств;
- крупнейшие поставщики по сумме.

Экспорт: XLSX и PDF.

Acceptance:

- режимы `all`, `accruals`, `payments` совпадают с текущим отчетом по выплатам;
- отчет показывает открытые обязательства без ручной сверки;
- выгрузки пригодны для передачи председателю или бухгалтеру.

## Технический Контракт Для Каждого Слота

1. Request DTO должен содержать период, фильтры, `limit`, search и массивы идентификаторов.
2. Response DTO должен содержать totals, rows, `rowCount` и признак обрезанной выдачи.
3. Backend endpoint размещается в `ReportsController`.
4. Расчеты размещаются в `ReportService`.
5. Для тяжелых выборок использовать PostgreSQL aggregation, server pagination или virtualization.
6. Frontend должен показывать loading, error, empty state, состояние без прав и счетчик показанных строк.
7. Экспорт XLSX/PDF должен использовать те же фильтры, что экранная таблица.
8. Все файлы и документация должны пройти UTF-8/no BOM проверку.
