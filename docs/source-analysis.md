## Техническое задание

ПО нужно для финансового учета деятельности гаражно-строительного кооператива.

Основные разделы:

- Контрагенты: группы "Гаражи" и "Поставщики и персонал", редактируемые группы поставщиков.
- Тарифы: тарифы для владельцев гаражей и для поставщиков/персонала с датой вступления в силу.
- Платежи: поступления от владельцев и оплаты поставщикам/персоналу.
- Отчеты: консолидированный отчет, отчет по поступлениям, отчет по выплатам, до 3 дополнительных отчетов, экспорт в XLSX и PDF.

Справочники и сущности:

- Виды оплат/поступлений: вода, мусор, электроэнергия со ступенчатым тарифом, членский взнос, целевой взнос, вступительный взнос, подключение канализации, подключение электричества, штраф, предписание.
- Виды выплат/расходов: электроэнергия, вывоз мусора, водоснабжение, банковские услуги, юридические услуги, зарплата персоналу, прочие выплаты, штрафы.
- Гаражи: номер, количество человек, количество этажей, владелец, стартовые счетчики воды и электричества, комментарий.
- Владельцы: гараж, идентификатор владельца, ФИО, телефон, адрес, текстовое поле по счетчикам.
- Поставщики: наименование, ИНН, юр. адрес, контактное лицо, телефон, почта, стартовый баланс, комментарий; обязательное поле только наименование.

Поступления:

- Поиск по номеру гаража и ФИО владельца, лучше единая строка с быстрыми подсказками.
- Шапка показывает данные гаража/владельца, итоговый баланс, кнопки просмотра платежей и добавления платежа.
- Таблица помесячная: месяц, счетчики воды и электричества, разницы, сумма начислений, сумма оплат, задолженность.
- Каждый первый день месяца добавляется новый месяц с регулярными платежами.
- Регулярные платежи: вода ежемесячно по счетчику, электричество ежемесячно по ступенчатому тарифу, мусор ежемесячно по количеству людей, членский и целевой взнос ежегодно, остальные начисления вручную.
- Если счетчики не заполнены, связанные поля подсвечиваются мягким красным, но доступные начисления считаются.
- Двойной клик по сумме начислений открывает разбивку начислений по услугам, добавление нерегулярных начислений и редактирование автосумм с обязательным комментарием.
- По электричеству нужно предупреждение, если предыдущий платеж был раньше, чем месяц назад; предупреждение не должно блокировать внесение платежа.

Оплаты:

- Выбор поставщика или отдела для зарплаты.
- Шапка показывает данные поставщика и кнопки просмотра/добавления платежа.
- Таблица: месяц, сумма начислений, сумма выплат, задолженность, комментарий.
- По умолчанию первого числа добавляются фиксированные начисления по вывозу мусора и зарплатам отделов.
- Комментарии по платежам контрагентам и комментарий за месяц являются разными данными.

Отчеты:

- Консолидированный отчет: месяц, сумма поступлений, сумма выплат, разница; первая строка с итогами, сортировка от последнего месяца к первому, фильтр по периоду ММ.ГГГГ.
- Отчет по поступлениям: дата, гараж, владелец, услуга, сумма начисления, поступление, комментарий; фильтры по датам, гаражам, владельцам, услугам, выбор нескольких значений, режим "только начисления" или "только поступления", итоговая первая строка.
- Отчет по выплатам: дата, поставщик, сумма начисления, сумма выплаты, комментарий; фильтры по датам и поставщикам, выбор нескольких поставщиков, режим "только начисления" или "только выплаты", итоговая первая строка.

## Старая Access-БД

Файл `ГСК 01.10.2018.accdb` обнаружен и имеет формат `Standard ACE DB`.

Ограничение текущей машины:

- доступные ODBC-драйверы читают старый `.mdb`, но не открыли `.accdb`;
- COM-объект Access присутствует, но вызов `OpenCurrentDatabase` падает из-за ошибки загрузки type library;
- полноценную схему и данные нужно снимать через установленный ACE-драйвер для `.accdb`, Access repair/compact или конвертацию в промежуточный формат.

Из бинарного анализа файла видны ориентиры старой схемы:

- таблицы/объекты: `garage`, `Vladelci`, `vidoplati`, `VidViplat`;
- поля/идентификаторы: `GARAGENUMBER`, `OwnerID`, `FAMILIA`, `IMYA`, `OTCHESTVO`, `TELEFHONE`, `ADRES`, `CHELOVEK`, `ETAZHI`, `PREDZNACHENIE`, `NOVZNACHENIE`, `RAZNICA`, `OPLACHENO`, `SUMMA`, `DATAPLATEZ`, `DATAVIPLATI`, `CENAPLATEZ`, `KOPLATE`, `ANOTACIA`.

Вывод: ТЗ соответствует структуре старой базы, но импорт нужно проектировать как отдельный проверяемый процесс с диагностикой, картой соответствия и отчетом о переносе.

## Дополнительные правила из заметок 30.06.2026

- Все рабочие страницы должны давать понятный возврат в главное меню.
- Тарифы должны поддерживать включаемую пороговую тарификацию, настраиваемое количество порогов и единицу измерения; по умолчанию пороговая тарификация относится к электроэнергии, единица - кВт.
- Просрочка возникает не в момент отрицательного баланса, а после установленного срока переноса долга: для воды, электричества и мусора после месяца погашения плюс 30 дней; для годовых членских/целевых взносов после даты оплаты плюс 30 дней; для наружного освещения после окончания года.
- Формы услуг должны показывать зависимые переключатели постепенно: нельзя предлагать пороговую тарификацию, если оплата не по счетчику.
- Нерегулярные платежи относятся к фонду прочих оплат; целевые сборы создаются отдельной формой, по умолчанию для всех гаражей, но с возможностью выбора участников.
- В счетчиках показание вводится прямо в годовой таблице, а расход справа считается автоматически как текущее минус предыдущее или стартовое значение из карточки гаража.
- Поступления работают от поиска гаража и месячной таблицы: платеж вводится прямо в ячейку, Enter очищает ввод и пересчитывает оплачено, задолженность и кассу.
- Выплаты работают по месячным циклам, учитывают кассу и банковский счет, проверяют наличие денег на счете при оплате поставщикам и переносят остатки/долги в новый месяц.
- Отчеты предназначены для просмотра, фильтрации, сортировки и экспорта без ручного ввода: отдельно нужны отчеты по гаражам, выплатам, оплатам, оплатам из кассы, сдаче кассы в банк и сборам.
- Управление фондами распределяет собранные средства между фондами; изъятие отправляет сумму в нераспределенные средства, пополнение фонда ограничено доступной нераспределенной суммой.
- По каждому рабочему объекту должна вестись история изменений, чтобы можно было проверить, что и почему насчитал бухгалтер: гараж, владелец, поставщик/сотрудник, тариф, услуга, счетчик, начисление, платеж, выплата, сбор, фонд, отчетная корректировка и импортная строка должны иметь проверяемый audit-след.
- История объекта должна показывать кто изменил запись, когда, какое действие выполнил, основание/комментарий, старые и новые значения ключевых полей, связанный месяц/гараж/контрагента/документ и результат пересчета денег. Для финансовых исправлений комментарий обязателен; физическое удаление заменяется отменой или архивированием с причиной.
