# Receipt Printing Scenario Decision Template

Этот шаблон нужен перед реализацией фактической печати чеков или квитанций. Он помогает выбрать юридический и технический сценарий, не подменяя решение заказчика предположением разработчика.

## Source Context

- `docs/source-analysis.md`: по чекам нужно уточнить, требуется ли фискальный чек по 54-ФЗ или внутренняя квитанция/чек; ориентиры оборудования - АТОЛ 30Ф/30Ф+ и Эвотор 5i.
- `docs/receipt-printing-design.md`: архитектура уже разделяет внутреннюю квитанцию и фискальный чек через `IReceiptPrintingAdapter`.
- `docs/archive/project-roadmap.md`: Stage 10 оставляет выбор сценария, оборудования, обязанного лица и реквизитов в `[decision]`.

## Decision Owner

- Дата решения:
- Кто подтверждает юридический сценарий:
- Кто подтверждает бухгалтерские реквизиты:
- Кто подтверждает оборудование или отсутствие оборудования:
- Нужна ли консультация бухгалтера/юриста по 54-ФЗ: да/нет

## Option A: Internal Receipt

Выбрать, если кооперативу нужна внутренняя квитанция для учета без фискализации.

Подтвердить:

- [ ] Внутренняя квитанция юридически достаточна для выбранных операций.
- [ ] Фискальный чек по 54-ФЗ не требуется для этих операций или будет оформляться отдельно вне GarageBalance.
- [ ] Утверждены обязательные реквизиты: название ГСК, ИНН/реквизиты, номер гаража, владелец, период, назначение, сумма, дата, номер документа, подписи/ответственный.
- [ ] Подтвержден формат: PDF, печать через браузер, локальный принтер или сохранение файла.
- [ ] Подтверждено, нужна ли печать копии с отметкой `КОПИЯ`.

Результат для roadmap:

- `Реализовать печать внутренней квитанции, если выбран нефискальный сценарий` можно перевести в работу.
- Фискальную интеграцию оставить `[decision]` или `[ ]` до отдельного решения.

## Option B: Fiscal Receipt

Выбрать, если нужен фискальный чек или интеграция с кассовым оборудованием.

Подтвердить:

- [ ] Операции GarageBalance действительно требуют фискализации.
- [ ] Выбрано оборудование: АТОЛ 30Ф/30Ф+, Эвотор 5i или другое.
- [ ] Понятен способ подключения: локальный драйвер, HTTP-сервис, облачный сервис, файл обмена или ручной операторский шаг.
- [ ] Подтверждены обязательные реквизиты чека, налоговый режим, предмет расчета, способ расчета, ставки/признаки и правила возврата/коррекции.
- [ ] Известно, где хранить безопасные идентификаторы устройства и кто отвечает за кассу.
- [ ] Есть тестовое устройство, эмулятор или безопасный тестовый контур.

Результат для roadmap:

- `Реализовать интеграцию с выбранным фискальным оборудованием, если нужен фискальный чек` можно перевести в работу.
- Внутреннюю квитанцию оставить отдельным fallback/документом только если заказчик это подтвердил.

## Decision Record

Выбрать один вариант:

- [ ] Internal receipt only.
- [ ] Fiscal receipt only.
- [ ] Internal receipt now, fiscal receipt later.
- [ ] Fiscal receipt plus internal copy/reporting form.
- [ ] Decision postponed.

Обоснование:

```
Кратко записать, почему выбран этот вариант, какие операции покрыты, какие операции исключены и какие риски остаются.
```

## Safe Data Rules

- Не переносить в Git параметры кассы, токены, пароли, fiscal storage identifiers, полные ответы драйвера, реальные телефоны, адреса, банковские реквизиты и персональные документы.
- Для ошибок устройства фиксировать только безопасный статус, код, короткое сообщение, correlation id и время.
- Секреты оборудования хранить только через protected settings, environment/deployment secrets или локальные настройки вне репозитория.

## Close Conditions

Пункт `Выбрать сценарий: фискальный чек по 54-ФЗ или внутренняя квитанция/чек` можно закрыть как `[x]` только после заполненного decision record с ответственным и выбранным вариантом. До этого пункты реализации внутренней квитанции, фискальной интеграции и тестов выбранного сценария должны оставаться незакрытыми.
