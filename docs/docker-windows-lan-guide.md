# GarageBalance на Windows: установка, порты и доступ из локальной сети

Эта инструкция предназначена для администратора, который устанавливает готовый Docker-релиз GarageBalance на один Windows-компьютер и при необходимости открывает доступ другим компьютерам той же локальной сети.

## 1. Как устроена установка

Основной компьютер выполняет роль сервера и должен оставаться включенным во время работы пользователей. На нем запускаются Docker Desktop, PostgreSQL, API и веб-интерфейс GarageBalance. Остальные компьютеры ничего не устанавливают и открывают приложение в браузере по локальному адресу сервера.

По умолчанию все внешние порты привязаны к `127.0.0.1`, поэтому приложение доступно только на основном компьютере. Для локальной сети наружу открывается только порт веб-интерфейса. API и PostgreSQL должны оставаться на `127.0.0.1`.

## 2. Требования

На основном компьютере нужны:

- Windows 10/11 x64;
- установленный и запущенный Docker Desktop;
- режим Linux containers;
- интернет только для предварительного скачивания Docker Desktop и ZIP; сама установка из распакованного архива работает без сети;
- постоянная папка, например `C:\GarageBalance`;
- желательно закрепленный локальный IP-адрес.

Git, .NET SDK, Node.js и отдельный PostgreSQL конечному пользователю не нужны.

## 3. Первая установка

1. Откройте [GarageBalance Releases](https://github.com/Xsenus/GarageBalance/releases).
2. Скачайте `GarageBalance-Docker-ВЕРСИЯ.zip` и соседний файл `.sha256` при необходимости проверки архива.
3. Полностью распакуйте ZIP в постоянную папку, например `C:\GarageBalance`.
4. Запустите Docker Desktop и дождитесь готовности Docker Engine.
5. Запустите `start.cmd` из распакованной папки.
6. Дождитесь сообщения об успешном запуске и открытия браузера.
7. При первом открытии создайте администратора GarageBalance.

Первый запуск автоматически:

- создает локальный `.env`;
- генерирует уникальные пароли и ключи;
- импортирует готовые API/frontend images из ZIP;
- импортирует PostgreSQL image из того же ZIP;
- создает постоянные Docker volumes;
- применяет миграции базы;
- проверяет `/health`.

Стандартный адрес после установки: <http://127.0.0.1:5173>.

## 4. Изменение порта на основном компьютере

Файл `.env` появляется после первого запуска `start.cmd`. Чтобы изменить веб-порт:

1. Остановите приложение через `stop.cmd`.
2. Откройте PowerShell и выполните:

```powershell
notepad C:\GarageBalance\.env
```

3. Найдите параметры:

```dotenv
FRONTEND_PORT=5173
FRONTEND_ORIGIN=http://127.0.0.1:5173
```

4. Например, для порта `8080` задайте:

```dotenv
FRONTEND_PORT=8080
FRONTEND_ORIGIN=http://127.0.0.1:8080
```

5. Сохраните `.env` и запустите `start.cmd`.

Новый адрес: <http://127.0.0.1:8080>.

Проверить, занят ли порт:

```powershell
Get-NetTCPConnection -LocalPort 8080 -ErrorAction SilentlyContinue
```

Если команда показывает слушающий процесс, выберите другой порт. Внешние `API_PORT=5080` и `POSTGRES_PORT=5432` обычно менять не требуется. При конфликте их можно заменить на другие свободные порты, не изменяя внутренние порты в `docker-compose.yml`.

## 5. Доступ с других компьютеров локальной сети

### 5.1. Найдите правильный IP сервера

На основном компьютере выполните:

```powershell
ipconfig
```

Найдите IPv4-адрес активного Ethernet или Wi-Fi адаптера, например `192.168.1.50`. Не используйте адрес Docker, WSL, VPN или отключенного адаптера.

Для стабильной работы закрепите этот адрес за компьютером через DHCP reservation в настройках роутера. Если адрес изменится, пользователям потребуется новый URL и корректировка `FRONTEND_ORIGIN`.

### 5.2. Настройте `.env`

Остановите приложение через `stop.cmd`, откройте `C:\GarageBalance\.env` и задайте, например:

```dotenv
FRONTEND_BIND_ADDRESS=0.0.0.0
FRONTEND_PORT=8080
FRONTEND_ORIGIN=http://192.168.1.50:8080

API_BIND_ADDRESS=127.0.0.1
API_PORT=5080

POSTGRES_BIND_ADDRESS=127.0.0.1
POSTGRES_PORT=5432
```

Правила:

- `FRONTEND_BIND_ADDRESS=0.0.0.0` разрешает веб-подключения через сетевые адаптеры сервера;
- `FRONTEND_ORIGIN` должен содержать фактический IP или DNS-имя и выбранный frontend-порт;
- в браузере нельзя использовать `0.0.0.0` — открывайте фактический IP сервера;
- `API_BIND_ADDRESS` и `POSTGRES_BIND_ADDRESS` оставляйте равными `127.0.0.1`;
- не публикуйте PostgreSQL в локальную сеть без отдельной защищенной административной необходимости.

Сохраните `.env` и запустите `start.cmd`. На сервере приложение теперь открывается по адресу `http://192.168.1.50:8080`.

### 5.3. Разрешите порт в Windows Firewall

Сначала убедитесь, что домашняя/рабочая сеть Windows имеет профиль `Private`. Не меняйте корпоративные политики сети без согласования с системным администратором.

Откройте PowerShell от имени администратора и выполните:

```powershell
New-NetFirewallRule `
  -DisplayName "GarageBalance 8080" `
  -Direction Inbound `
  -Action Allow `
  -Protocol TCP `
  -LocalPort 8080 `
  -Profile Private `
  -RemoteAddress LocalSubnet
```

Так порт разрешается только для частного профиля и устройств локальной подсети. Если выбран другой порт, замените `8080` в имени и `LocalPort`.

Проверить правило:

```powershell
Get-NetFirewallRule -DisplayName "GarageBalance 8080"
```

Удалить правило:

```powershell
Remove-NetFirewallRule -DisplayName "GarageBalance 8080"
```

### 5.4. Проверьте подключение с другого ПК

На другом компьютере той же сети выполните:

```powershell
Test-NetConnection 192.168.1.50 -Port 8080
```

Ожидаемый результат: `TcpTestSucceeded : True`.

Затем откройте в браузере:

```text
http://192.168.1.50:8080
```

Можно попробовать имя компьютера, например `http://GARAGE-PC:8080`; имя показывает команда `hostname`. Фиксированный IP или локальная DNS-запись обычно надежнее автоматического поиска Windows-имени.

## 6. Что резервируется

`backup.cmd` создает и проверяет PostgreSQL backup в папке `C:\GarageBalance\backups`. Он не копирует `.env` и volume ключей защиты.

При обычном обновлении `.env` сохраняется: релизный ZIP содержит только `.env.example`. Тем не менее для восстановления после потери компьютера сделайте отдельную защищенную копию `.env`:

```powershell
Set-Location C:\GarageBalance
$stamp = Get-Date -Format "yyyyMMdd_HHmmss"
Copy-Item .env "backups\.env_$stamp.backup"
```

В `.env` находятся секреты. Не отправляйте его посторонним, не прикладывайте к диагностике и не публикуйте в Git.

Ключи защиты не входят в PostgreSQL backup. После первичной настройки и после смены ключей сохраните volume отдельно:

```powershell
docker run --rm `
  -v garagebalance_data-protection-keys:/keys:ro `
  -v C:/GarageBalance/backups:/backup `
  alpine:3.22 `
  tar -czf /backup/data-protection-keys.tar.gz -C /keys .
```

Для полноценного аварийного комплекта нужны:

- свежий проверенный `.pgdump`;
- защищенная копия `.env`;
- архив `data-protection-keys.tar.gz`;
- установочный ZIP соответствующей версии.

Храните копию на другом физическом носителе или в защищенном хранилище. Подробная проверка и восстановление базы описаны в [руководстве по PostgreSQL backup](postgres-backup-restore.md).

## 7. Обновление

1. Запустите `backup.cmd` и дождитесь проверки копии.
2. Сохраните отдельную копию `.env` и ключей защиты по разделу 6.
3. Скачайте ZIP новой версии.
4. Не удаляйте существующую папку, `.env`, `backups`, `logs` и Docker volumes.
5. Распакуйте новый ZIP поверх `C:\GarageBalance` с заменой файлов программы.
6. Запустите `update.cmd`.
7. Дождитесь создания обязательного предобновляющего backup и успешного `/health`.
8. Проверьте вход, гаражи, платежи, отчеты и раздел «Что нового».

Сетевые параметры сохраняются в `.env` и после обновления остаются прежними.

## 8. Штатные команды

- `start.cmd` — первая установка или повторный запуск;
- `stop.cmd` — безопасная остановка без удаления данных;
- `update.cmd` — обновление только после проверенного backup;
- `backup.cmd` — ручной PostgreSQL backup;
- `diagnostics.cmd` — технический отчет без `.env` и секретов.

Не выполняйте `docker compose down -v`: параметр `-v` удаляет volumes базы и ключей защиты.

## 9. Диагностика локальной сети

Если приложение работает на сервере, но не открывается с другого ПК, проверьте:

1. Docker Desktop запущен.
2. В `.env` указано `FRONTEND_BIND_ADDRESS=0.0.0.0`.
3. Используется актуальный IP сервера.
4. В Windows Firewall разрешен выбранный frontend-порт.
5. Оба компьютера находятся в одной подсети.
6. Wi-Fi роутер не использует гостевую сеть или client isolation.
7. VPN не меняет маршрут локального трафика.
8. `Test-NetConnection IP PORT` возвращает `True`.

Состояние контейнеров на сервере:

```powershell
Set-Location C:\GarageBalance
docker compose --project-name garagebalance --env-file .env -f docker-compose.yml ps
```

Для безопасного технического отчета запустите `diagnostics.cmd`.

## 10. Доступ через интернет

Не пробрасывайте frontend, API или PostgreSQL напрямую на роутере. Для доступа через интернет требуются HTTPS, домен и reverse proxy либо защищенный VPN. PostgreSQL должен оставаться недоступным из интернета. Для серверного развертывания используйте [инструкцию VPS, домена и TLS](vps-deployment-checklist.md).
