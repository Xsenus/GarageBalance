# Checklist передачи проекта в Git

Документ используется только когда пользователь отдельно разрешит публикацию проекта в удаленный Git-репозиторий. До этого момента разрешены локальные коммиты, но запрещены `git push`, создание pull request и публикация архива проекта.

## 1. Что Проверить До Публикации

1. Убедиться, что рабочий каталог находится в `C:\Users\ilel\Desktop\GarageBalance`.
2. Выполнить `git status --short` и разобрать все изменения.
3. Проверить, что в индекс попадают только файлы текущей задачи.
4. Проверить, что `.gitignore` защищает реальные базы, дампы, backup, локальные настройки и приватные импорты.
5. Выполнить `infrastructure/scripts/verify-package-privacy.ps1`, чтобы проверить tracked/untracked Git-кандидаты перед публикацией или архивом поставки.
6. Выполнить `git ls-files --others --exclude-standard` и убедиться, что приватные файлы не стали untracked-кандидатами для публикации.
7. Проверить, что история roadmap заполнена: что сделано, зачем, какие проверки пройдены и почему live-проверки могли быть невозможны.
8. Проверить, что видимое пользователю изменение добавлено в `backend/GarageBalance.Api/AppReleases/releases.json`.

## 2. Что Нельзя Публиковать

В Git нельзя добавлять:

- реальные `.env`, `appsettings.Local.json`, production secrets и JWT-ключи;
- Access-БД `.accdb` и `.mdb`;
- PostgreSQL backup и dump: `.pgdump`, `.dump`, `.backup`, `.sql.gz`;
- релизные артефакты миграций и временные SQL-файлы в `artifacts/`;
- папки `artifacts/`, `backups/`, `dumps/`, `private-imports/`, `imports/private/`, `imports/raw/`;
- платежные документы, договоры, счета, чеки и персональные выгрузки;
- реальные файлы отчетов XLSX/PDF, созданные для заказчика;
- логи с персональными, финансовыми или секретными данными;
- пароли от VPS, root-доступы, токены 1C Fresh, данные фискального оборудования.

Если файл нужен только для диагностики, его надо обезличить и сохранить вне репозитория.

## 3. Обязательные Проверки Перед Commit

Перед каждым локальным коммитом выполнить:

1. Профильный тест измененной области, если он есть.
2. `dotnet test`.
3. `npm run test -- --runInBand` в папке `frontend`.
4. `npm run build` в папке `frontend`.
5. `npm run lint` в папке `frontend`.
6. `dotnet format --verify-no-changes`.
7. `infrastructure/scripts/generate-migration-script.ps1` для проверки миграционной цепочки и SQL-артефакта.
8. `git diff --check`.
9. JSON-проверку `backend/GarageBalance.Api/AppReleases/releases.json`.
10. Строгую проверку UTF-8/no BOM для измененных `.md`, `.json`, `.cs`, `.ts`, `.tsx`, `.css`, `.html`, `.yml`, `.yaml`, `.conf`, `.ps1` и Dockerfile.
11. `infrastructure/scripts/verify-package-privacy.ps1`.

Если локальный PostgreSQL, `psql` или Docker недоступны, это нужно явно записать в roadmap и финальный ответ. Нельзя писать, что live DB/Docker smoke выполнен, если `postgresTcp=False`, `psql=False` или `docker=False`.

## 4. Локальный Commit

1. Выполнить `git diff --stat`.
2. Просмотреть `git diff` по измененным файлам.
3. Добавить в индекс только файлы текущей задачи через `git add -- <files>`.
4. Повторно выполнить `git status --short`.
5. Сделать коммит с русским сообщением, например `Добавить checklist передачи проекта в Git`.
6. Проверить `git log -1 --oneline`.

Локальный коммит разрешен в рамках разработки. `git push` запрещен до отдельной команды пользователя.

## 5. Подготовка К Удаленному Репозиторию

Когда пользователь разрешит публикацию:

1. Уточнить имя удаленного репозитория и ветку.
2. Проверить, что remote не указывает на чужой проект: `git remote -v`.
3. Если remote еще не добавлен, добавить его только после подтверждения URL.
4. Проверить ветку и при необходимости создать рабочую ветку с префиксом `feature/`.
5. Еще раз выполнить проверки из раздела "Обязательные Проверки Перед Commit".
6. Выполнить `git push` только после явного разрешения пользователя.
7. Если нужен pull request, создать его только после отдельного подтверждения.

## 6. Контроль После Публикации

1. Проверить, что удаленный репозиторий открывается.
2. Проверить, что приватные файлы не появились в web-интерфейсе Git.
3. Проверить, что CI запускает backend/frontend проверки.
4. Если CI упал, не делать force push без анализа причины.
5. Если приватный файл все-таки попал в Git, немедленно остановить работу, сообщить пользователю и готовить ротацию секретов.

## 7. Короткая Команда Для Финальной Самопроверки

Перед публикацией пройти этот список:

- `git status --short` чистый или содержит только ожидаемые файлы;
- `.gitignore` защищает Access-БД, дампы, backup и secrets;
- `releases.json` валиден и содержит запись для видимого изменения;
- roadmap содержит историю выполненной работы;
- backend/frontend тесты пройдены;
- UTF-8/no BOM проверен;
- PostgreSQL/Docker статус честно зафиксирован;
- `git push` выполняется только после прямой команды пользователя.
