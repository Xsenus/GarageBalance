# Meter Reading Operator Scenario Verification

Этот документ фиксирует evidence для закрытия верхнеуровневого сценария оператора по показаниям счетчиков.

## Закрытая Область

- [x] Оператор вводит и изменяет показания воды и электроэнергии через backend API с audit-историей.
- [x] Расход рассчитывается от предыдущего активного или стартового показания гаража.
- [x] Годовая таблица загружает сохраненные показания и записывает новые значения через API.
- [x] Годовая таблица загружает гаражи серверными страницами по 25 строк, получает только компактные значения выбранного года и показывает скелетон во время запроса.
- [x] Пропущенные показания и разрывы последовательности видны оператору.
- [x] Регулярные начисления поддерживают тарифы `meter_water` и `meter_electricity`.
- [x] Изменение показания атомарно пересчитывает связанное неоплаченное начисление по сохраненному тарифу; полностью или частично оплаченное начисление возвращает конфликт без изменения данных.
- [x] Месячная ведомость гаража показывает показание, расход, начисление, оплату и задолженность.
- [x] Сводный отчет показывает количество показаний по месяцам и гаражам.

## Проверочное Покрытие

- [x] `FinanceServiceTests` проверяет расчет расхода, округление, разрыв истории, дубли, отмену, восстановление и начисления по тарифам счетчиков.
- [x] `FinanceServiceTests` и `PostgreSqlManualWaterMeterReadingIntegrationTests` явно проверяют стартовое значение, последовательные месяцы, одинаковое показание с нулевым расходом, дробное показание и запрет уменьшения.
- [x] `PostgreSqlManualWaterMeterReadingIntegrationTests` и frontend workflow объединяют ручную воду с пустой текущей строкой: до ввода сохраняются независимый членский взнос и нулевая водная сумма, после ручного сохранения исчезают красная подсветка и доступная подсказка, появляются подтвержденные расход и сумма воды, а членский взнос не меняется.
- [x] `FinanceServiceTests`, `PostgreSqlMeterReadingConcurrencyIntegrationTests` и frontend workflow проверяют неизменность прошлой/будущей записи после обычной попытки, отсутствие ложного audit и разрешенную историческую корректировку с отдельным правом, обязательной причиной и единственным audit-событием.
- [x] `FinanceServiceTests` и `PostgreSqlMeterReadingConcurrencyIntegrationTests` проверяют успешный пересчет начисления, запрет после частичной оплаты и откат показания при ошибке сохранения начисления.
- [x] `FinanceServiceTests.GetGarageIncomeWorksheetAsync_BuildsRowsFromAccrualsPaymentsAndMeters` проверяет месячную ведомость с показанием, расходом, начислением, оплатой и долгом.
- [x] `ReportServiceTests.GetConsolidatedReportAsync_ReturnsMonthlyTotalsAndGarageRows` проверяет счетчики показаний в месячных и гаражных строках отчета.
- [x] `App.test.tsx` проверяет годовую таблицу, сохранение показаний, рассчитанный расход и подсветку гаражей без показаний.
- [x] `FinanceServiceTests` и `FinanceControllerTests` проверяют фильтр года/типа, исключение архивных гаражей, полноту значений страницы, валидацию и передачу пагинации.
- [x] `ProjectWideRoadmapStatusTests.MeterReadingOperatorScenarioIsCompleteWhenYearTableWorksheetAccrualsDebtAndReportsAreCovered` удерживает roadmap-status вместе с evidence.

- [x] Схема дополнена индексом активных показаний по типу, месяцу и гаражу; миграция проверяется на локальной PostgreSQL и service-тестами SQLite.

## История выполнения

- 2026-07-18 — PostgreSQL- и frontend-сценарии объединяют ручную воду, отсутствие текущего показания, красную доступную подсветку и независимый членский взнос до и после подтвержденного ручного ввода.
- 2026-07-18 — усилено покрытие границ редактирования: запрещенные обычные изменения прошлого и будущего месяца оставляют PostgreSQL-значения без изменений и не создают обычный audit, а разрешенная историческая корректировка изменяет только прошлую запись и сохраняет причину в отдельном событии.
- 2026-07-18 — матрица граничных значений закреплена отдельным service-тестом одинакового показания и единой PostgreSQL-цепочкой от стартового значения через последовательные месяцы, нулевой и дробный расход до отклоненного уменьшения.
- 2026-07-13 — форма переведена с двух общих запросов (гаражи с балансами и расширенный список показаний) на один компактный годовой endpoint с server-side пагинацией. Это устраняет обрезание результата до 500 показаний, сокращает объем ответа и ограничивает таблицу 300 редактируемыми ячейками на странице вместо тысяч.
