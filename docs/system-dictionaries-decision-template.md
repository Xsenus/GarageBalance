# System Dictionaries Decision Template

Этот документ помогает согласовать, какие справочники GarageBalance являются системными, какие полностью управляются пользователем и какие изменения допустимы для каждой категории. Решение должен подтвердить заказчик или ответственный бухгалтер; шаблон не меняет действующие правила приложения сам по себе.

## Current Behavior

Сейчас признак `IsSystem` есть у групп поставщиков, видов поступлений, видов выплат и фондов. Для системных групп и видов операций backend запрещает изменение и архивирование. Пользовательские записи этих справочников можно создавать, изменять, архивировать и восстанавливать с проверкой дублей и audit-событиями.

Владельцы, гаражи, поставщики, контакты поставщиков, сотрудники, отделы, тарифы, настройки услуг, нерегулярные платежи и сборы не имеют признака `IsSystem`. Для исторически используемых записей применяется soft-archive, а не физическое удаление.

Migration `DefaultAccountingTypes` создает системные виды поступлений:

- `water` - Вода;
- `trash` - Мусор;
- `electricity` - Электроэнергия;
- `membership` - Членский взнос;
- `target` - Целевой взнос;
- `entry` - Вступительный взнос;
- `connection` - Подключения;
- `penalty` - Штраф;
- `notice` - Предписание.

Системные виды выплат:

- `electricity` - Электроэнергия;
- `trash_removal` - Вывоз мусора;
- `water_supply` - Водоснабжение;
- `bank` - Банковские расходы;
- `legal` - Юридические расходы;
- `salary` - Зарплата;
- `other` - Прочие расходы;
- `penalty` - Штрафы.

Коды `water`, `electricity`, `membership`, `target`, `entry`, `connection` и `salary` уже участвуют в расчетах, подборе тарифа или генерации начислений. Их нельзя менять без совместимой миграции данных и тестов.

## Classification Matrix

Для каждой строки выбрать один класс:

- `system_immutable` - запись обязательна, имя и код не меняются, архивирование запрещено;
- `system_named` - стабильный код обязателен, отображаемое имя можно менять с audit;
- `user_managed` - пользователь создает, меняет, архивирует и восстанавливает запись;
- `historical_only` - новые записи не создаются, существующие доступны только для истории;
- `not_a_dictionary` - объект является рабочим документом/операцией и регулируется отдельными правилами.

## Allowed Operations Matrix

Для каждого выбранного класса подтвердить операции:

| Класс | Создать | Переименовать | Изменить код | Архивировать | Восстановить | Hard delete |
| --- | --- | --- | --- | --- | --- | --- |
| `system_immutable` | только migration/seed | нет | нет | нет | только repair/migration | нет |
| `system_named` | только migration/seed | да, с audit | нет | `<decision>` | `<decision>` | нет |
| `user_managed` | да | да | `<decision>` | да, с причиной | да, с причиной | нет для использованных записей |
| `historical_only` | нет | нет | нет | уже скрыт | нет | нет |

Отдельно подтвердить:

- [ ] может ли администратор создать новую запись со своим кодом;
- [ ] можно ли менять код пользовательской записи после первого начисления или платежа;
- [ ] нужно ли блокировать архивирование записи, связанной с активным тарифом или услугой;
- [ ] должны ли системные записи отображаться первыми и иметь явную метку `Системный`;
- [ ] можно ли восстанавливать пользовательскую запись при конфликте имени или кода;
- [ ] кто имеет право выполнять изменение и архивирование: `dictionaries.write`, `tariffs.manage` или отдельное permission.

## Stable Code Rules

- Стабильный `Code` используется как машинный идентификатор, а `Name` - как пользовательская подпись.
- Для `system_immutable` и `system_named` код уникален, не пуст и не меняется через UI/API.
- Изменение уже используемого кода допускается только отдельной миграцией с анализом финансовых правил, импорта, отчетов и интеграций.
- Пользовательские записи без машинной логики могут иметь nullable code только после явного подтверждения.
- Дубли активного имени и кода должны обрабатываться одинаково в API, импорте и восстановлении из архива.

## Access Import Rules

Выбрать правило сопоставления старой Access-БД:

- [ ] системные записи сопоставляются сначала по стабильному коду, затем по утвержденному alias-списку;
- [ ] совпадение только по имени не перезаписывает системную запись автоматически;
- [ ] неизвестный вид создается как `user_managed` после preview;
- [ ] конфликт кода/имени отправляется в quarantine и требует решения;
- [ ] импорт не меняет `IsSystem`, код и имя защищенной записи;
- [ ] reconciliation report отдельно показывает mapped, created, skipped и quarantined dictionary rows.

## Permissions And Audit

- Создание, изменение, архивирование, восстановление и смена классификации должны проверяться на backend.
- Для опасного действия обязательны actor, время, причина, старые и новые значения, тип справочника и идентификатор записи.
- Смена `IsSystem` не должна быть обычным полем пользовательской формы; если она разрешена, нужен отдельный подтверждаемый workflow.
- Frontend скрывает недоступные команды, но backend остается источником запрета.
- Ошибка должна объяснять, что защищено: запись целиком, стабильный код или только архивирование.

## UI Expectations

- Системные и пользовательские записи имеют явную текстовую метку, доступную не только по цвету.
- Для защищенной записи недоступные команды не показываются или disabled с понятной причиной.
- Списки поддерживают фильтр `Все / Системные / Пользовательские / Архивные`, если объем данных этого требует.
- Диалог архивирования показывает влияние на тарифы, услуги и исторические операции.
- Конфликт восстановления сообщает, какая активная запись мешает восстановлению, без раскрытия чувствительных данных.

## Implementation Impact

После утверждения решения проверить необходимость:

- migration для новых `IsSystem`, unique code или classification-полей;
- изменения DTO и validation API;
- отдельных permission/policy;
- изменения `DictionaryService`, import mapping и reconciliation;
- UI-фильтров, меток и disabled states;
- audit metadata и end-user release note;
- backend, frontend, migration и PostgreSQL integration tests.

## Safe Data Rules

- Не включать в decision record реальные ФИО, телефоны, адреса, реквизиты или суммы.
- Не прикладывать Access-файлы, дампы, `.env`, credentials и raw import rows.
- Для примеров использовать только вымышленные названия и коды.
