# Защита Данных И Сроки Хранения

Документ фиксирует, какие данные GarageBalance считаются чувствительными, какие требуют повышенной защиты и как хранить импортные файлы/логи. Это входной контракт для будущего шифрования настроек интеграций, импорта Access, audit-журнала, backup и передачи проекта.

## Классы Данных

| Класс | Примеры | Защита | Где используется |
| --- | --- | --- | --- |
| Публичные служебные | версия приложения, health-status без деталей, названия системных ролей | Можно показывать авторизованным пользователям по правам | `GET /health`, `GET /api/app-releases`, `GET /api/users/roles` |
| Рабочие данные ГСК | номера гаражей, тарифы, виды поступлений/выплат, суммы начислений, платежи, поставщики | Доступ только по permissions, audit действий, no-store cache для API | справочники, финансы, отчеты |
| Персональные данные | ФИО владельцев, телефоны, адреса, email пользователей, контактные лица поставщиков | Минимизация выдачи, masking в audit/logs, доступ по ролям, не отправлять в issue/чат без обезличивания | `owners`, `app_users`, `suppliers` |
| Импортные данные | `.accdb`/`.mdb`, dry-run report, SHA-256 файла, quarantine/error bucket, строки с ошибками | Хранить вне Git, ограниченный доступ, checksum вместо лишнего копирования, удалить raw-файлы после приемки | `access_import_runs`, private import folders |
| Секреты и токены | JWT signing key, connection string с паролем, будущие токены 1C Fresh, ключи фискального оборудования/API | Только env/user-secrets/deployment secrets; хранение чувствительных настроек и токенов интеграций шифровать перед записью в БД | `Jwt__SigningKey`, `.env`, будущие integration settings |
| Диагностические данные | application logs, nginx/systemd/docker logs, stack traces, export logs | Перед передачей маскировать email, token, secret, api_key, password, Bearer, длинные номера; не прикладывать дампы целиком | troubleshooting, deploy, support |

## Поля Повышенной Защиты

- `Owner.LastName`, `Owner.FirstName`, `Owner.MiddleName`, `Owner.Phone`, `Owner.Address`, `Owner.MeterNotes` - персональные данные владельцев и заметки по счетчикам.
- `AppUser.Email`, `AppUser.DisplayName`, `AppUser.PasswordHash` - учетные данные пользователей; `PasswordHash` никогда не возвращать через API и не логировать.
- `Supplier.ContactPerson`, `Supplier.Phone`, `Supplier.Email`, `Supplier.Inn` - контакты и реквизиты поставщика; выдача только пользователям с рабочими правами.
- `FinancialOperation.Amount`, `DocumentNumber`, `Comment`, `CancelReason` - финансовые сведения и возможные персональные комментарии; видимость по permissions, запись действий в audit.
- `Accrual.Comment`, `SupplierAccrual.Comment`, `MeterReading.Comment` - комментарии могут содержать ручные пояснения оператора, поэтому считаются чувствительными.
- `AccessImportRun.OriginalFileName`, `ContentSha256`, `ReportJson` - данные импорта; raw-файл не хранить в Git и не отдавать через публичные endpoints.
- `Jwt:SigningKey`, `Jwt__SigningKey`, `ConnectionStrings:Default`, будущие `OneCFresh` токены и настройки фискального принтера - секреты контура, хранить только вне репозитория; при сохранении в БД использовать encryption at rest.

## Правила Хранения Импорта

1. Исходные `.accdb`/`.mdb` использовать только из локальных приватных папок `private-imports/`, `imports/private/` или `imports/raw/`; эти пути запрещены для Git.
2. Оригинал Access-БД не изменять. Для экспериментов создавать рабочую копию и фиксировать SHA-256, имя файла, размер и дату dry-run.
3. Raw-файлы импорта хранить до завершения сверки и приемки переноса, но не дольше 30 календарных дней после финального успешного импорта.
4. Dry-run reports в `access_import_runs.ReportJson` хранить до завершения проекта и гарантийного периода приемки, затем оставить только итоговую сверку, checksum, дату и статус.
5. Quarantine/error bucket хранить до исправления блокеров и подписания сверочного отчета; строки с персональными данными не отправлять наружу без обезличивания.
6. Backup перед импортом хранить по backup-регламенту, но не использовать как замену raw-файлу Access и не передавать в Git/чат.

## Правила Логов

- Application, docker, nginx и systemd logs для диагностики хранить локально до закрытия инцидента, затем удалить или обезличить.
- Перед отправкой разработчику прогонять ручную проверку: нет `.env`, connection string с паролем, JWT, Bearer token, `password`, `secret`, `api_key`, полной Access-БД, дампа PostgreSQL и неанонимизированных списков владельцев.
- В audit-журнале хранить бизнес-событие, пользователя, сущность и краткое описание, но не хранить пароли, токены, секреты, полные файлы импорта и содержимое дампов.
- CSV-экспорт audit должен использовать то же masking-правило, что экранный audit.

## Требования К Будущему Шифрованию

1. Секреты интеграций 1C Fresh и фискального оборудования должны храниться в БД только в зашифрованном виде.
2. Ключ шифрования не хранить в Git и не отдавать через API; задавать через env/user-secrets/deployment secrets.
3. Rotation ключей должен иметь план: новый ключ для новых записей, безопасная миграция старых значений и audit-событие администраторского действия.
4. Логи ошибок шифрования не должны содержать plaintext секретов.
5. Тесты должны проверять, что секретное значение не возвращается из API, не попадает в audit summary и не пишется в release/debug output.
6. Для интеграционных секретов использовать `ISensitiveDataProtector` и `IIntegrationSecretSettingsService`: сохранять в `integration_secret_settings` только значения с префиксом `gb:protected:v1:`, разделять токены по purpose (`OneCFresh.Token`, `FiscalPrinter.Token`) и настраивать `DataProtection:KeysPath` на стендах, где ключи должны переживать перезапуск.

## Acceptance-Критерии

- В roadmap закрыты пункты определения чувствительных полей и сроков хранения импортных данных.
- README ссылается на этот документ рядом с правилами приватных файлов.
- `verify-package-privacy.ps1` продолжает блокировать `.env`, `.accdb`, `.mdb`, дампы, backup и private/raw import-папки.
- Новые интеграции и import features перед реализацией сверяются с этим документом.
