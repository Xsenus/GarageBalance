# Troubleshooting guide

Практическая инструкция для администратора и разработчика при типовых сбоях GarageBalance. Перед любыми действиями с рабочими данными сначала создайте backup и не отправляйте наружу секреты, персональные данные, финансовые детали, `.env`, `.accdb`, `.mdb`, `.pgdump` и дампы.

Если интерфейс показывает код ошибки, сохраните его и время. Администратор формирует пакет через `Настройки` → `Диагностика`; состав пакета и правила передачи описаны в `docs/diagnostic-logging-guide.md`. Не отправляйте вместо него `.env`, backup или базу данных.

## 1. Сайт не открывается

Проверить локальный ПК:

```powershell
curl -fsS http://127.0.0.1:5080/health
docker compose ps
docker compose logs --tail=200 api
docker compose logs --tail=200 frontend
```

Проверить VPS:

```bash
curl -fsS https://sgk.blagodaty.ru/health
systemctl status garagebalance-staging.service
journalctl -u garagebalance-staging.service -n 200 --no-pager
nginx -t
tail -n 200 /var/log/nginx/error.log
```

- [ ] Если health endpoint отвечает, но страница не открывается, проверить frontend/nginx и cache policy `Cache-Control: no-store`.
- [ ] Если health endpoint не отвечает, проверить backend service/container.
- [ ] Если nginx config менялся, выполнять `systemctl reload nginx` только после успешного `nginx -t`.
- [ ] Если обновление сломало стенд, использовать `docs/version-update-checklist.md` и rollback.

## 2. База недоступна

Локальный ПК:

```powershell
docker compose ps postgres
docker compose logs --tail=200 postgres
Test-NetConnection 127.0.0.1 -Port 5432
```

Без Docker:

```powershell
pg_isready -h 127.0.0.1 -p 5432 -d garagebalance_local
psql --host=127.0.0.1 --port=5432 --username=garagebalance_local --dbname=garagebalance_local --command="select 1;"
```

- [ ] Проверить connection string `ConnectionStrings__DefaultConnection`.
- [ ] Проверить, что пароль базы не изменился отдельно от конфигурации API.
- [ ] Не пересоздавать рабочую базу до проверки backup.
- [ ] Перед восстановлением выполнить `restore-postgres.ps1` в `garagebalance_restore_check`.
- [ ] Если миграции не применились, сначала сгенерировать `dotnet tool run dotnet-ef migrations script --idempotent`.

## 3. Импорт Access упал

- [ ] Не менять исходный `.accdb`/`.mdb`.
- [ ] Создать backup PostgreSQL через `backup-postgres.ps1`.
- [ ] Запустить dry-run импорта и скачать JSON-отчет.
- [ ] Проверить предупреждения про ACE-драйвер, Microsoft Access, OLE-сигнатуру, размер файла и SHA-256.
- [ ] Проверить, что файл не лежит в Git и не отправляется в чат без отдельного согласования.
- [ ] Для разработчика подготовить только безопасный JSON-отчет dry-run, имя файла, размер, SHA-256 и список checks без персональных данных.

## 4. Отчет не сходится

- [ ] Проверить период отчета: даты и месяцы не должны быть перепутаны местами.
- [ ] Проверить режим строк: `all`, `accruals` или `payments`.
- [ ] Проверить фильтры по гаражам, владельцам, поставщикам, видам поступлений и видам выплат.
- [ ] Проверить стартовые балансы гаражей и поставщиков.
- [ ] Проверить, что регулярные начисления за месяц не были пропущены или запущены повторно.
- [ ] Проверить показания счетчиков и предупреждения о разрыве истории.
- [ ] Сравнить экранный отчет с XLSX/PDF-выгрузкой за тот же период.
- [ ] Для диагностики передавать разработчику период, фильтры, тип отчета и обезличенные итоговые суммы, а не полную финансовую таблицу.

## 5. Пользователь не может войти

- [ ] Проверить, активен ли пользователь в разделе "Пользователи".
- [ ] Проверить роль и permissions через матрицу ролей.
- [ ] Проверить историю изменений на `auth.login_failed`, `auth.login_inactive`, `auth.login_rate_limited`.
- [ ] При подозрении на подбор пароля выполнить администраторский сброс пароля.
- [ ] Не отправлять текущий или новый пароль в открытый чат.
- [ ] Если вход временно ограничен, дождаться окончания rate limit или проверить причину серии ошибок.

## 6. "Что нового" не загружается

- [ ] Проверить `GET /api/app-releases` после входа.
- [ ] Проверить JSON-валидность `backend/GarageBalance.Api/AppReleases/releases.json`.
- [ ] Проверить, что новая запись релиза стоит в начале массива.
- [ ] Проверить, что пользователь авторизован.
- [ ] Если JSON поврежден, восстановить файл из последнего commit и повторить `npm run build`/`dotnet test`.

## 7. Принтер или 1C Fresh недоступны

Интеграции с 1C Fresh и печатью чеков относятся к следующему этапу. До реализации адаптеров использовать такой порядок:

- [ ] Зафиксировать время ошибки и действие пользователя.
- [ ] Проверить, это фискальный сценарий или внутренняя квитанция.
- [ ] Проверить настройки выбранного адаптера, когда он будет добавлен.
- [ ] Не повторять массовую отправку документов без idempotency/журнала интеграции.
- [ ] Для 1C Fresh подготовить тестовый контур, направление обмена и список документов/операций.
- [ ] Для печати подготовить модель устройства, драйвер, тип подключения и пример проблемной квитанции без персональных данных.

## 8. Что можно отправлять разработчику

Можно:

- [ ] Версию приложения и commit.
- [ ] Точное время ошибки.
- [ ] Роль пользователя без ФИО.
- [ ] Название раздела и действие.
- [ ] Health-check результат.
- [ ] Коды ошибок API без токенов.
- [ ] Замаскированный audit-фрагмент.
- [ ] JSON dry-run отчета импорта без исходной Access-БД и персональных данных.
- [ ] Обезличенные фильтры отчета и итоговые суммы.

Нельзя:

- [ ] `.env`, connection string, JWT secret, пароли и токены.
- [ ] `.accdb`, `.mdb`, `.pgdump`, `.sql.gz` и дампы PostgreSQL.
- [ ] Полные платежные таблицы, реквизиты, ФИО, телефоны и email без отдельного согласия.
- [ ] Скриншоты с персональными или финансовыми данными.

## 9. Минимальный порядок безопасной диагностики

- [ ] Создать backup, если проблема связана с данными.
- [ ] Проверить `/health`.
- [ ] Проверить актуальность версии в "Что нового".
- [ ] Проверить историю изменений.
- [ ] Проверить логи без секретов.
- [ ] Зафиксировать найденные действия в roadmap history или журнале работ.
