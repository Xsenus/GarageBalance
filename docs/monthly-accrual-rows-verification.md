# Monthly Accrual Rows Verification

Этот документ фиксирует evidence для закрытия roadmap-пункта Stage 6 "Реализовать помесячные строки начислений".

## Закрытая Область

- [x] Backend хранит начисления владельцев в `Accruals` по гаражу, виду начисления и месяцу учета.
- [x] Backend хранит начисления поставщиков в `SupplierAccruals` по поставщику, виду выплаты, документу и месяцу учета.
- [x] Ручные начисления требуют комментарий.
- [x] Регулярные начисления нормализуют месяц до первого числа.
- [x] Повторный запуск регулярных начислений за тот же месяц не создает дубли.
- [x] Backend автоматически проверяет текущий месяц при запуске и далее по расписанию, формируя отсутствующие регулярные начисления по активному каталогу услуг.
- [x] Автоматизация использует рабочий часовой пояс Новосибирска и после перезапуска догоняет пропущенный запуск без дублей.
- [x] Массовая генерация загружает существующие начисления и показания счётчиков пакетно, без N+1-запросов по каждому гаражу.
- [x] Отмена начисления требует причину, исключает строку из активных итогов и пишет audit.
- [x] Восстановление отмененного начисления проверяет конфликт с активной заменой.
- [x] Частичные уникальные индексы защищают активные начисления от дублей и разрешают замену после отмены.
- [x] UI показывает таблицы "Последние начисления" и "Последние начисления поставщикам".
- [x] UI поддерживает добавление, редактирование, отмену, восстановление и просмотр разбивки начисления.
- [x] UI поддерживает перенос задолженности владельца на следующий месяц через `createDebtTransfer`.

- [x] `FinanceServiceTests` проверяет автоматический запуск в текущем рабочем месяце и безопасный повторный запуск.

- [x] `FinanceControllerTests` проверяет HTTP-surface для начислений, отмены, восстановления, генерации регулярных начислений и переноса долга.

## История выполнения

- 2026-07-14: подтверждено, что прежняя реализация запускала каталог регулярных начислений только вручную. Добавлена фоновая проверка текущего месяца при старте приложения и каждые шесть часов по времени Новосибирска; повторный запуск защищен существующей проверкой дублей. Ручная команда теперь объясняет, что источником служит каталог регулярных услуг, показывает итог созданных и пропущенных строк и направляет в «Тарифы и сборы», если каталог не настроен. Проверка закреплена backend- и React-тестами; после реализации остаётся ручная приемка на заполненном рабочем каталоге услуг.
- 2026-07-14: повторный аудит обнаружил N+1-проверки дублей и показаний в массовой генерации. Существующие начисления и показания месяца теперь загружаются пакетно; тест на 200 гаражах подтверждает постоянное число SELECT-запросов как при первом, так и при повторном запуске. Что остаётся: приемка автоматизации на заполненном каталоге после публикации ветки.
