# Порядок добавления новых отчетов

Документ применяется после первых трех базовых отчетов: консолидированный отчет, отчет по поступлениям и отчет по выплатам. Каждый новый отчет должен добавляться как проверяемая функция, а не как разовая таблица в интерфейсе.

## 1. Заявка И Границы Отчета

1. Зафиксировать, для кого отчет нужен: бухгалтер, председатель, администратор, заказчик или внешний контрагент.
2. Записать бизнес-вопрос, на который отчет отвечает.
3. Определить период: день, месяц, квартал, год или произвольный диапазон.
4. Определить, какие суммы считаются по дате начисления, а какие по дате фактической оплаты.
5. Зафиксировать формат дат в фильтрах: `ДД.ММ.ГГГГ` для дневных периодов и `ММ.ГГГГ` для месячных срезов.
6. Определить, нужен ли отчет в ежемесячном цикле и должен ли он попадать в `docs/monthly-cycle-checklist.md`.

## 2. Данные И Инварианты

1. Перечислить источники данных: владельцы, гаражи, тарифы, показания счетчиков, начисления, поступления, выплаты, начисления поставщикам, импорт Access.
2. Проверить, есть ли у записей источник и внешний идентификатор/хэш, если данные пришли из импорта.
3. Описать, как отчет учитывает отмененные операции и ручные корректировки.
4. Описать, как отчет показывает задолженность, переплату, начислено, оплачено и разницу.
5. Проверить, что отчет не требует прямого SQL-удаления или ручного переноса из Access.
6. Если данных не хватает, сначала добавить доменную модель, EF Core migration и audit-события, а потом сам отчет.

## 3. Backend Контракт

1. Создать request DTO с фильтрами, `limit`, optional search и массивами идентификаторов, если нужны мультивыборы.
2. Создать response DTO с итогами, строками, `rowCount` и признаком ограниченной выдачи, если таблица может быть большой.
3. Добавить метод в `ReportService`; контроллер должен оставаться тонким.
4. Добавить endpoint в `ReportsController` под правом `reports.read`.
5. Для экспорта добавить отдельные endpoints XLSX/PDF, если отчет нужен вне экрана.
6. Если экспорт записывает событие в историю изменений, endpoint выгрузки должен использовать `POST`, а не безопасный `GET`.
7. Для ошибок использовать общий `ProblemDetails` contract: `validation_failed`, `unauthorized`, `forbidden`, `internal_error`.
8. Не переносить тяжелую агрегацию в память приложения, если ее можно выполнить PostgreSQL-запросом.

## 4. Производительность

1. Фильтрацию, сортировку и агрегацию выполнять на стороне PostgreSQL.
2. Для экранной таблицы использовать `limit`, `rowCount`, server pagination или virtualization.
3. Проверить поиск по индексируемым полям.
4. Не загружать полный объем строк только ради отображения первых строк.
5. Для XLSX/PDF повторно использовать те же фильтры, что экранный отчет.
6. После добавления тяжелого отчета проверить frontend bundle и не добавлять крупную библиотеку без записи в roadmap.

## 5. Экспорт XLSX/PDF

1. XLSX должен содержать лист строк и лист итогов.
2. PDF должен быть пригоден для печати и отправки заказчику без ручного копирования из интерфейса.
3. Имя файла должно следовать формату `garagebalance-{reportType}-{yyyyMMdd}-{yyyyMMdd}.{xlsx|pdf}` или другому формату, отдельно зафиксированному в roadmap.
4. Экспорт должен учитывать те же права, период, поиск, мультивыборы и режимы строк, что экранная версия.
5. Если отчет содержит персональные или финансовые данные, выгрузку нельзя хранить в Git.

## 6. Frontend

1. Добавить клиентский метод в reports API.
2. Добавить UI-фильтры с понятными labels и validation summary.
3. Показать loading, error, empty state и состояние без прав.
4. Для больших таблиц показывать счетчик отображенных строк и общий `rowCount`.
5. Добавить кнопки скачивания XLSX/PDF только пользователю с `reports.read`.
6. Сохранять фильтры в `sessionStorage`, если отчет нужен в регулярной работе.
7. Проверить адаптивность на ноутбуке и большом мониторе; мобильная версия не основная, но экран не должен разваливаться.

## 7. Тесты

Backend:

1. Unit/service tests для итогов, фильтров, дат, округления и пустых данных.
2. Controller tests для routes, status codes, DTO shape, validation, authorization и error contract.
3. EF/integration tests для mappings, PostgreSQL migration script и сложных запросов.
4. Export tests для XLSX/PDF content type, имени файла и совпадения фильтров с экранным отчетом.

Frontend:

1. React workflow test загрузки отчета.
2. Тест validation summary.
3. Тест loading/error/empty state.
4. Тест фильтров, мультивыборов и сохранения `sessionStorage`.
5. Тест скачивания XLSX/PDF.
6. Тест недоступности отчета без `reports.read`.

Общие проверки перед коммитом: `dotnet test`, `npm run test -- --runInBand`, `npm run build`, `npm run lint`, `dotnet format --verify-no-changes`, idempotent EF migration script, `git diff --check`, JSON validation для `releases.json` и UTF-8/no BOM.

## 8. Документация И Приемка

1. Обновить README, если появился новый endpoint, экспорт или новый сценарий работы.
2. Обновить `docs/user-guide.md`, если отчет виден пользователю.
3. Обновить `docs/monthly-cycle-checklist.md`, если отчет участвует в закрытии месяца.
4. Добавить запись в `backend/GarageBalance.Api/AppReleases/releases.json`.
5. Обновить историю `docs/project-roadmap.md`: что сделано, зачем, какие проверки прошли и был ли доступен PostgreSQL/Docker.
6. Если отчет входит в расширенную приемку, явно указать срок проверки и критерии.

## 9. Acceptance Checklist

- Есть понятный бизнес-вопрос отчета.
- Даты, суммы и источники данных зафиксированы.
- Endpoint закрыт правом `reports.read`.
- Экранная таблица не грузит лишние данные.
- XLSX/PDF используют те же фильтры, что экран.
- Есть backend и React tests.
- Есть запись в "Что нового".
- Roadmap history заполнена.
- UTF-8/no BOM проверен.
- Live DB/Docker smoke выполнен или честно указано, почему он невозможен.
