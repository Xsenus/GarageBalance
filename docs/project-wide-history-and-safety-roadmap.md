## Целевое Правило Для Всего Проекта

- `[ ]` Любое создание, изменение, архивирование, удаление, восстановление, отмена, импорт, начисление, платеж, настройка, изменение прав, изменение пользователя, ручная корректировка и интеграция должны создавать историю изменения.
- `[ ]` История должна храниться на backend и в PostgreSQL, а не только в React-state.
- `[ ]` История должна показывать минимум: дата и время, пользователь, раздел, тип объекта, идентификатор объекта, видимое название объекта, действие, поле/группа полей, старое значение, новое значение, причина/комментарий, связанный гараж/контрагент/месяц/документ при наличии.
- `[x]` В рабочих разделах не должно быть отдельных вкладок `История изменений`. История доступна через единую вкладку левого меню `История изменений`.
- `[ ]` В рабочих разделах допустимы только короткие ссылки на объект или действие, но не отдельные таблицы истории внутри каждого раздела.
- `[ ]` Удаление по умолчанию является soft delete / archive / cancel с audit-событием; физическое удаление запрещено для рабочих объектов без отдельного решения.
- `[~]` Для каждого soft-deleted объекта, который можно безопасно вернуть, должна быть кнопка `Вернуть` / `Восстановить`.
- `[ ]` Восстановление должно проверять конфликты уникальности и бизнес-ограничения, например номер гаража, название активного тарифа, документ платежа.
- `[ ]` Каждое опасное действие удаления/архивации/отмены должно открывать confirmation dialog.
- `[ ]` Каждое сохранение формы с изменениями должно открывать confirmation dialog с понятным списком `было -> стало`.
- `[ ]` Если изменений нет, форма закрывается или сохраняется без confirmation dialog.
- `[ ]` Все контролы проекта должны использовать единый стиль: кнопки, icon-only кнопки, input, textarea, select, date/month/year controls, checkbox, toggle, tabs, dialogs, tables, pagination, filters, empty/loading/error states.

## Решения И Допущения

- `[x]` Единый пользовательский экран истории будет отдельным разделом левого меню с названием `История изменений`. Существующий `Аудит` можно расширить и переименовать, чтобы не плодить два похожих раздела.
- `[x]` Для пользователей без права `audit.read` раздел истории не показывается и API истории недоступен.
- `[x]` В истории не показываются секреты, пароли, токены, полные персональные/банковские данные и другие чувствительные значения; применяются существующие правила маскирования.
- `[x]` Для frontend-прототипов история, которая сейчас хранится локально внутри раздела, считается временной. При подключении backend API она должна уйти в общий журнал.
- `[decision]` Нужно решить, как назвать левый пункт: `История изменений`, `Журнал изменений` или оставить `Аудит`. Предварительно использовать `История изменений`, потому что так просит заказчик.
- `[decision]` Нужно решить, какие отмененные финансовые операции можно именно восстановить, а какие нужно проводить обратной операцией. Предварительно: справочники и настройки восстанавливаются, платежи/начисления восстанавливаются только если это безопасно и разрешено бизнес-правилом, иначе создается обратная операция с audit-связью.
- `[decision]` Нужно решить срок хранения истории. Предварительно: бессрочно для рабочих данных ГСК.

## Риски И Вопросы

- `[decision]` Название левого пункта: `История изменений` или `Аудит`.
- `[decision]` Правило восстановления финансовых операций: restore исходной записи или обратная операция.
- `[decision]` Какие пользователи могут видеть старые/новые значения по чувствительным объектам.
- `[decision]` Нужна ли подпись/причина для обычных изменений всех объектов или только для финансовых/критичных.
- `[decision]` Нужно ли экспортировать историю сразу или позже.
- `[!]` Возможный риск: текущие frontend-прототипы содержат локальную историю. Нельзя считать это финальной историей до подключения backend.
- `[!]` Возможный риск: добавление confirmation ко всем формам может замедлить работу бухгалтера, поэтому no-op save и неопасные create-flow должны оставаться быстрыми.
- `[!]` Возможный риск: восстановление архивных объектов может конфликтовать с active-only unique indexes.
