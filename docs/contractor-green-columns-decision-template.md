# Согласование фильтров по зеленым колонкам контрагентов

Дата подготовки: 12.07.2026. Обновлено: 19.07.2026.
Статус: исходные зеленые колонки подтверждены прямым read-only разбором Open XML; правила реализованы для таблицы гаражей.

## Зачем Нужна Фиксация

В пояснительной записке указано: «Зеленые поля – предусмотреть сортировку и фильтр». Read-only разбор `xl/styles.xml`, shared strings и листа `Контрагенты` исходной книги подтвердил заливку `FF92D050` только у ячеек `A6:C6`. Персональные и финансовые строки при сверке не извлекались и не сохранялись.

До согласования не используются реальные строки Excel с ФИО, телефонами, адресами, суммами или иными персональными и финансовыми данными. Достаточно обезличенного скриншота заголовков или текстового списка колонок.

## Что Уже Работает

- независимый режим `Показать должников` для гаражей и поставщиков;
- сортировка видимых строк таблиц гаражей, поставщиков и персонала;
- поиск гаражей и поставщиков;
- запоминаемая ширина колонок и понятные пустые состояния;
- переход `Открыть финансовый отчет` для гаража, поставщика и сотрудника.

## Матрица Колонок

Для каждой зеленой колонки заполнить отдельную строку. Если колонка отсутствует в конкретной Excel-форме, оставить ее вне таблицы.

Утверждено 17.07.2026: фильтр каждой колонки определяется ее фактическим типом данных — текст, список, число/сумма, дата или признак да/нет. Специальное бизнес-правило допускается только после отдельного явного согласования; тип нельзя выбирать по цвету или отображаемому примеру значения.

| Таблица | Точный заголовок Excel | Поле/смысл | Фильтр | Сортировка | Пустые значения | Пример без реальных данных | Решение |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Гаражи | `Номер` | строковый номер гаража | содержит, без учета регистра | возр./убыв. | поле обязательно; пустой фильтр не ограничивает выборку | `А-20` | `[x]` |
| Гаражи | `Количество человек` | целое неотрицательное число | минимум / максимум / диапазон | возр./убыв. | пустая граница диапазона не применяется | `2–4` | `[x]` |
| Гаражи | `Количество этажей` | целое неотрицательное число | минимум / максимум / диапазон | возр./убыв. | пустая граница диапазона не применяется | `1–2` | `[x]` |

У заголовков таблиц поставщиков и персонала зеленая заливка в исходной книге отсутствует, поэтому дополнительные фильтры для них по этому требованию не вводятся.

Допустимые варианты фильтра для согласования:

- текст: содержит / начинается с / точное совпадение;
- список: одно значение / несколько значений;
- число или сумма: минимум / максимум / диапазон;
- дата: с / по / диапазон;
- признак: да / нет / все;
- специальное бизнес-правило: описать формулой и привести обезличенные граничные примеры.

## Обязательные Решения

- `[x]` Точный список подтвержден по стилям исходного XLSX: три колонки гаражей; у поставщиков и персонала зеленых заголовков нет.
- `[x]` Для каждой подтвержденной колонки использовать фильтр ее фактического типа данных; конкретный тип и допустимые значения фиксируются в заполненной матрице.
- `[x]` Сортировка сохраняет существующие направления; пустое поле или граница означает отсутствие соответствующего условия.
- `[x]` Фильтры применяются на сервере до подсчета, сортировки и пагинации по всей выборке.
- `[x]` Режим `Показать должников` пересекается с фильтрами зеленых колонок и не сбрасывает их.
- `[x]` Введенные и примененные значения сохраняются при смене вкладки в рамках открытого экрана, но сбрасываются после перезагрузки или нового входа.
- `[x]` Фильтры влияют только на таблицу контрагентов; финансовый отчет и экспорт имеют собственные явно выбранные фильтры.
- `[x]` Названия, кнопки сброса и пустое состояние реализованы на русском языке.

## Техническая Проверка После Решения

- сопоставить каждую колонку с существующим DTO/полем БД или отдельно согласовать новую модель и migration;
- добавить server-side filtering/sorting/pagination и индексы для растущих таблиц;
- сохранить backend permissions `dictionaries.read` и не раскрывать архивные/чувствительные данные без разрешения;
- добавить frontend controls, сброс фильтров, loading/empty/error/permission states;
- покрыть backend и React-тестами каждую колонку, сочетание с должниками, пагинацию, пустые значения и переход в финансовый отчет;
- проверить PostgreSQL query plan на реалистичном объеме после доступности локальной или приемочной БД.

## Критерий Согласования

Решение считается подтвержденным, когда:

- заполнена матрица без `[заполнить]` и `[decision]` для выбранных колонок;
- приложен обезличенный скриншот заголовков или текстовый перечень исходных Excel-колонок;
- для каждой колонки определены фильтр, сортировка и пустые значения;
- зафиксированы совместимость с должниками, сохранение состояния и влияние на отчет/экспорт;
- заказчик явно подтвердил правила в переписке или приемочном документе.

Решение реализовано отдельным пользовательским срезом и получает запись в "Что нового". Архивный roadmap и его исторический статус не изменяются.
