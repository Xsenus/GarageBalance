# Roles And First Administrator Decision Template

Этот шаблон нужен для ручного подтверждения фактических ролей ГСК и первого администратора перед финальной приемкой на реальных пользователях. Он не заменяет уже реализованные роли, permissions и bootstrap первого администратора, а фиксирует бизнес-решение заказчика.

## Source Context

- `docs/project-roadmap.md`: Stage 1 уже содержит проектирование пользователей, bootstrap первого администратора и развитие granular permissions.
- `docs/admin-operations-guide.md`: администратор создает первого пользователя при пустой базе и управляет сотрудниками.
- `docs/local-pc-install-checklist.md`: локальная установка требует создания первого администратора или входа существующим пользователем.
- Acceptance matrix требует сверки реальных ролей заказчика перед финальным закрытием.

## First Administrator

Заполнять без паролей, токенов, паспортных данных, телефонов, полных адресов и лишних персональных данных.

- [ ] Первый администратор определен заказчиком.
- [ ] У первого администратора есть право заводить пользователей, назначать роли и сбрасывать пароль.
- [ ] Канал передачи первичного доступа согласован вне Git и вне публичных документов.
- [ ] Подтверждено, кто отвечает за хранение доступа после локальной установки.
- [ ] Подтверждено, что в Git не попали реальные email, пароли, телефоны, адреса и документы.

## Backup Administrator

- [ ] Назначен резервный администратор или подтверждено, что резервного администратора пока нет.
- [ ] Описан порядок восстановления доступа, если первый администратор недоступен.
- [ ] Подтверждено, кто может запросить сброс доступа.
- [ ] Подтверждено, что отключение последнего администратора запрещено.

## Role Matrix

Заполнить фактические роли кооператива и отметить, какие системные роли GarageBalance им соответствуют.

| Фактическая роль ГСК | Системная роль | Основные действия | Ограничения | Кто назначает |
| --- | --- | --- | --- | --- |
| Председатель/администратор | Administrator | Пользователи, настройки, справочники, отчеты, аудит | По решению заказчика | |
| Бухгалтер/оператор | Accountant/Operator | Платежи, начисления, счетчики, отчеты | Без управления пользователями, если не подтверждено | |
| Просмотр отчетов | ReportViewer | Просмотр и экспорт отчетов | Без записи платежей и справочников | |
| Ограниченная роль | Custom/ограниченная | Только согласованные разделы | Все остальное запрещено | |

## Permissions Acceptance

- [ ] Подтвержден доступ к пользователям и ролям.
- [ ] Подтвержден доступ к справочникам.
- [ ] Подтвержден доступ к тарифам.
- [ ] Подтвержден доступ к платежам и начислениям.
- [ ] Подтвержден доступ к отчетам и экспорту.
- [ ] Подтвержден доступ к импорту Access.
- [ ] Подтвержден доступ к настройкам, интеграциям, печати и audit-журналу.
- [ ] Подтверждены действия, которые должны быть доступны только администратору.

## Do Not Store

Не записывать в этот файл и не коммитить:

- пароли, временные коды, seed-фразы, токены, cookie и JWT;
- реальные телефоны, полные адреса, паспортные данные и банковские реквизиты;
- полные реальные email сотрудников, если они не предназначены для публичной документации;
- скриншоты с персональными или финансовыми данными;
- дампы БД, `.accdb`, `.mdb`, backup-файлы и приватные import folders.

## Close Conditions

Открытый вопрос `Нужно подтвердить реальные роли пользователей и кто будет первым администратором` можно закрыть как `[x]` только после заполненного decision record, подтвержденного первого администратора, согласованной role matrix и проверки доступа на реальных ролях без секретов и персональных данных в Git.
