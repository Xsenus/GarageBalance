# Сбор и передача ошибок GarageBalance

Система сохраняет технические ошибки так, чтобы администратор мог получить диагностический пакет у пользователя и передать его разработчику без копирования базы данных. Журнал не заменяет резервные копии и не содержит их.

## Что происходит при ошибке

- Каждый запрос получает безопасный код `X-Error-ID`. При внутренней ошибке этот же код показывается пользователю и записывается в журнал.
- Ошибки React, `window.error` и необработанные ошибки Promise после входа передаются в защищенный API с тем же кодом, который виден в интерфейсе.
- Сервер пишет события уровня `Error` и выше в построчные JSON-файлы. Файлы меняются ежедневно и при достижении лимита размера.
- Перед записью маскируются Bearer-токены, значения password/token/secret/apiKey, email и телефонные номера. Длина каждого сообщения ограничена.
- В диагностический ZIP входят только управляемые файлы ошибок и технический manifest. `.env`, база PostgreSQL, backup, ключи защиты и пользовательские документы не включаются.

Автоматическое маскирование снижает риск, но передавать пакет следует только доверенному специалисту по защищенному каналу.

## Действия пользователя и администратора

1. Пользователь записывает показанный код ошибки, время, раздел и последнее действие. Пароль, токен, `.env`, базу и лишние персональные данные передавать не нужно.
2. Администратор с правом `users.manage` открывает `Настройки` → `Диагностика`.
3. Проверяет состояние и время последней ошибки, затем нажимает `Скачать диагностический пакет`.
4. Передает ZIP доверенному специалисту вместе с кодом и временем. Создание пакета фиксируется в истории изменений без его содержимого.

Пользователь без административного права не видит вкладку и не может получить статус или ZIP. При отключенном журнале выгрузка недоступна.

## Docker

В `.env` задаются:

```dotenv
LOG_HOST_PATH=C:/GarageBalance/Logs
DIAGNOSTIC_LOGGING_ENABLED=true
DIAGNOSTIC_LOGGING_RETENTION_DAYS=14
DIAGNOSTIC_LOGGING_MAX_FILE_SIZE_MB=10
DIAGNOSTIC_PACKAGE_DAYS=7
DIAGNOSTIC_PACKAGE_MAX_SIZE_MB=20
```

Для Linux используйте, например, `/opt/garagebalance/logs`. Папка подключается в API как `/logs`, поэтому обычное обновление контейнера журнал не стирает. Ротация удаляет старые управляемые файлы автоматически. Не добавляйте ZIP и JSONL в Git.

Проверка состояния без просмотра содержимого:

```powershell
docker compose exec api sh -c "ls -lh /logs"
docker compose logs --tail=100 api
```

## VPS без Docker

Сценарий публикации создает закрытый каталог `/opt/garagebalance-staging/logs` и при отсутствии значений добавляет параметры `DiagnosticLogging__*` в `/etc/garagebalance-staging.env`. Каталог расположен вне заменяемой папки API и переживает обновление.

```bash
sudo ls -lh /opt/garagebalance-staging/logs
sudo systemctl status garagebalance-staging.service --no-pager
```

Не прикладывайте полный `journalctl`, `.env` или дамп БД без отдельной проверки: они могут содержать больше данных, чем ограниченный пакет.

## Если журнал не записывается

1. Проверить свободное место и права записи каталога.
2. Проверить безопасную причину последней ошибки записи во вкладке `Диагностика`.
3. Для Docker проверить `LOG_HOST_PATH` и подключение `/logs` командой `docker compose config`.
4. После исправления воспроизвести проблему и убедиться, что время последней записи изменилось.

Не отключайте маскирование и не увеличивайте лимиты без необходимости. Для долгого хранения перенесите нужный ZIP в защищенное хранилище с ограниченным доступом.
